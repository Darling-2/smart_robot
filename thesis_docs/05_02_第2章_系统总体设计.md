# 第2章 系统总体设计

## 2.1 需求分析

### 2.1.1 功能需求

本系统以室内走廊巡检为应用场景，通过任务与需求分析，归纳出以下五项核心功能需求：

**（1）室内环境地图构建**：系统需具备在未知室内环境中自主驱动采集激光雷达点云、实时构建二维占据栅格地图并完成回环修正的能力。地图作为后续导航与定位的基础，须真实反映走廊的几何轮廓、障碍物分布和通行区域。

**（2）多点顺序自主巡检导航**：系统需支持预设多个有序巡检点（Waypoints）并自主规划路径依次到达，导航过程中须能绕行静态障碍物（如杂物、临时放置的物品），并对动态行人进行减速回避，在无法通行时触发恢复行为（原地等待/重新规划）。

**（3）设施状态视觉检测**：系统到达每个巡检点后，需触发一次目标检测推理，对指定设施的当前状态（如门的开/关状态）进行识别与分类，输出类别标签与置信度。

**（4）巡检结果记录与告警**：系统需将每次检测结果以时间戳格式写入日志文件；当检测结果为"异常"（如门应关未关）时，需通过终端日志或预留告警接口输出警示信息。

**（5）系统自主性与鲁棒性**：除初始遥控建图外，系统须无需人工干预完成自主巡检；当导航路径被临时障碍物完全封堵时，应具备等待或重新规划的恢复能力，而不是永久卡死。

### 2.1.2 性能指标

根据功能需求与实际工程约束，制定以下可量化性能指标，作为后续实验验证的评估基准：

| 性能指标 | 目标值 | 测试方法 |
|:---|:---:|:---|
| SLAM建图关键尺寸误差 | ≤ 5% | 卷尺测量走廊真实宽度与地图像素折算宽度对比 |
| AMCL 定位误差（平移） | ≤ ±10 cm | 固定参考点多次返回误差统计 |
| 导航到点成功率（无障碍场景） | ≥ 90% | 20次独立任务统计 |
| 导航到点成功率（静态障碍场景） | ≥ 85% | 设置固定障碍物，10次独立任务统计 |
| YOLOv8n 检测 mAP@0.5 | ≥ 80% | 验证集（约120张）评估 |
| 树莓派5 YOLOv8n 推理帧率 | ≥ 3 FPS | ultralytics PyTorch CPU 推理耗时测量，配合 max_fps 节流 |
| 跨板通信（ZMQ）目标-结果往返延迟 | ≤ 20 ms | 发送端打点，回执端回传打点，100次统计 |
| 跨板通信（ZMQ）消息丢失率 | ≤ 1% | 连续发送100条目标，统计有效回执数 |

### 2.1.3 约束条件

系统设计须在以下约束条件下进行：

- **传感器约束**：仅使用 EAI 单线激光雷达（无三维点云）与 720p USB 单目摄像头（无深度），无 IMU 或 GPS，环境几何约束较弱；
- **算力约束**：主控 A（香橙派3LTS）仅 2GB RAM，RK3566 CPU 主频 1.8GHz；主控 B（树莓派5）4GB RAM，BCM2712 CPU 主频 2.4GHz，无 GPU 加速；
- **环境约束**：系统设计面向室内结构化走廊场景，地图规模适中（典型走廊长 20~50m），无需户外定位能力；
- **成本约束**：全系统硬件成本控制在 ¥2000 以内，不使用 Jetson 等高算力 AI 加速平台。

## 2.2 系统总体架构

### 2.2.1 三层异构架构设计

根据上述需求分析，本系统采用**"执行层—感知层—决策层"三层异构架构**，如图2-1所示。三层在功能分工上相互独立，在数据流上通过明确定义的 ROS 话题接口与跨板业务消息接口耦合，便于各层的独立调试与替换升级。

>
> **图 2-1**　系统三层异构软件架构总览

系统三层架构如下文所述（详细关系参见图2-1）：

```
决策层（树莓派5 / ROS2 Jazzy）：巡检任务调度 → 下发导航目标(nav_goal) → 触发检测 → 日志/告警
        ↕
感知层（树莓派5 / ROS2 Jazzy）：USB摄像头采集 | YOLOv8n 推理
        ↕  以太网 + ZeroMQ（nav_goal / nav_result）
执行层（香橙派3LTS / ROS1 Noetic）：slam_gmapping / amcl / move_base + base_control → STM32 → 差速电机 + EAI雷达
```

**执行层**（香橙派3LTS，ROS1 Noetic）承担与物理硬件直接交互与导航闭环控制的底层职责：`base_control.py` 节点通过串口（`/dev/ttyUSB0`，波特率115200）向 STM32 微控制器发送运动指令，STM32 负责电机 PWM 调速和编码器里程计计算，里程计数据回传后由 `base_control.py` 封装为 `/odom` 话题发布；`ydlidar_ros_driver` 节点通过 USB 口采集 EAI 单线激光雷达的 360° 扫描数据，以 `sensor_msgs/LaserScan` 消息格式发布 `/scan` 话题，频率约 10Hz。在此基础上，香橙派侧部署 `slam_gmapping` 完成建图、`amcl` 完成定位，并使用 `move_base` 输出 `/cmd_vel` 速度指令供底盘执行，实现自主导航闭环。

**感知层与决策层**（树莓派5，ROS2 Jazzy）运行巡检业务逻辑与视觉感知算法：720p USB 摄像头直连树莓派5，`v4l2_camera` 节点发布 `/image_raw` 话题供 YOLOv8n 检测节点订阅；巡检任务管理节点负责按照预设巡检点序列向香橙派侧下发导航目标，并在收到到点结果后触发一次检测与结果记录。

**跨板通信层**处于两板之间，在以太网链路上通过 ZeroMQ 传输最小化业务消息：树莓派下发巡检点对应的导航目标（`nav_goal`），香橙派在 `move_base` 任务结束后回传导航结果（`nav_result`）。该方式不做 ROS Topic 级透明桥接，树莓派5 端无需安装任何 ROS1 相关软件，实现关注点分离并降低跨板耦合。

上述接口设计遵循以下原则：**最小化跨板消息集合**（仅跨板传输 `nav_goal/nav_result`），减少跨板通信开销并避免 ROS1/ROS2 话题级兼容问题；与视觉检测相关的高带宽话题（`/image_raw`，720p 约 20MB/s）均在树莓派5本地流转，不经过以太网传输，避免带宽瓶颈；与导航闭环强相关的话题（`/scan`、`/odom`、`/tf`、`/cmd_vel`）全部在香橙派 ROS1 域内流转，提升系统整体稳定性。

## 2.3 硬件平台选型与集成

### 2.3.1 移动底盘与底层控制器

**NanoRobot 差速驱动底盘**是本系统的物理执行载体，实物如图2-2所示。底盘采用两轮差速驱动形式，机器人轮廓坐标为 $[[-0.09,\,-0.1],\,[0.04,\,-0.1],\,[0.04,\,0.1],\,[-0.09,\,0.1]]$（单位：m），尺寸约 13cm×20cm，可通过标准室内走廊（宽度≥0.5m）和普通门框。

> **【图 2-2 插入位置】**
> **内容**：NanoRobot 差速驱动底盘实物全貌照片（标注各硬件组件，如香橙派、树莓派5、激光雷达、USB摄像头、STM32控制板等）。
>
> **图 2-2**　NanoRobot 差速驱动底盘实物图

差速驱动运动学模型如下：设左轮线速度为 $v_L$，右轮线速度为 $v_R$，轮距为 $W$，则底盘线速度 $v$ 与角速度 $\omega$ 满足：

设左轮线速度为 $v_L$，右轮线速度为 $v_R$，轮距为 $W$，则底盘线速度 $v$ 与角速度 $\omega$ 满足：

$$v = \frac{v_R + v_L}{2}, \quad \omega = \frac{v_R - v_L}{W}$$

里程计通过对左右编码器计数积分，在 `odom` 坐标系下推算机器人位姿：

$$\begin{bmatrix} x_{t+1} \\ y_{t+1} \\ \theta_{t+1} \end{bmatrix} = \begin{bmatrix} x_t + v\Delta t \cos\theta_t \\ y_t + v\Delta t \sin\theta_t \\ \theta_t + \omega\Delta t \end{bmatrix}$$

底盘板载 **STM32F4 微控制器**负责电机 PWM 驱动与编码器采样，通过串口（USB CDC，波特率115200）与香橙派3LTS通信，接收速度指令并上报里程计。该控制层与 ROS 框架解耦，保证底层执行的实时性不受上层 ROS 调度抖动影响。

### 2.3.2 嵌入式处理器

**主控A——香橙派3LTS**（Orange Pi 3LTS）是面向 IoT 与轻量级嵌入式应用的 ARM 开发板，搭载 Rockchip RK3566 SoC（四核 Cortex-A55，主频 1.8GHz，Mali-G52 GPU），配备 2GB LPDDR4 RAM。本系统基于官方 Ubuntu 20.04 镜像，安装 ROS1 Noetic，专司底层执行与导航闭环：
- 运行 `base_control` 串口驱动节点，通过串口与 STM32 通信；
- 运行 `ydlidar_ros_driver`（ROS1 版本）采集激光雷达数据；
- 运行 `slam_gmapping`/`map_server`、`amcl` 与 `move_base` 完成建图、定位与自主导航；
- 运行 `zmq_nav_bridge`（ZeroMQ）节点，接收树莓派下发的导航目标并回传导航结果。

选用香橙派3LTS的理由：其 ROS1 Noetic 生态完整，EAI 激光雷达和底盘串口驱动均提供原生 ROS1 支持包，底层运动控制对算力要求不高（2GB RAM 已充裕），成本约 ¥200，具有良好的性价比。

**主控B——树莓派5（4GB）**（Raspberry Pi 5 4GB）是本系统计算核心，搭载 Broadcom BCM2712 SoC（四核 Cortex-A76，主频 2.4GHz，VideoCore VII GPU），配备 4GB LPDDR4X-4267 RAM，内置 PCIe 2.0 接口。本系统基于官方 Ubuntu 24.04 64位系统，安装 ROS2 Jazzy（LTS 版本），承担巡检业务与视觉计算：
- YOLOv8n 目标检测推理（ultralytics PyTorch，CPU 后端，配合 `max_fps` 参数节流）；
- 巡检任务调度节点（巡检点序列、目标下发、到点回执处理）；
- 720p USB 摄像头图像采集。

选用树莓派5的理由：BCM2712 的 Cortex-A76 核心单核算力相比前代树莓派4提升约2倍，4GB RAM 能够在 CPU 环境下稳定运行 YOLOv8n 推理与巡检调度逻辑，并预留后续扩展空间；ROS2 Jazzy 与 Ubuntu 24.04 的官方组合具有最长的支持周期（至2027年）。

### 2.3.3 传感器配置

**EAI YDLidar 单线激光雷达**通过 USB 数据线连接香橙派3LTS，提供 360° 水平扫描，测距范围 0.15m ~ 6.0m，扫描频率 10Hz，角度分辨率约 1°（每圈约360个采样点）。单线激光雷达仅获取水平平面内的距离信息，无法感知台阶、悬空障碍等三维特征，但对于室内走廊的二维地图构建和障碍物感知已完全满足需求，且成本（约¥200）远低于多线激光雷达。

激光雷达安装于底盘顶部，中心距地面高度约 15cm，水平安装（roll=0°，pitch=0°），相对于底盘中心的 TF 变换（`base_link → laser_link`）在 `robot_description` 的 URDF 文件中静态定义：

```xml
<!-- nanorobot_description.urdf 中激光雷达 TF 配置 -->
<joint name="laser_joint" type="fixed">
  <parent link="base_link"/>
  <child link="laser_link"/>
  <origin xyz="0.0 0.0 0.15" rpy="0 0 0"/>
</joint>
```

**720p USB 摄像头**直接连接树莓派5（`/dev/video0`），由 `v4l2_camera` 节点采集，分辨率 1280×720，帧率 30FPS（YOLOv8n 推理通过 `max_fps` 参数节流至约 5 FPS 异步消费）。摄像头安装于底盘前方竖直支架，镜头朝前方，覆盖距底盘约 0.5m ~ 2.0m 的检测区域。两个传感器的安装位置与坐标系关系如图2-3所示。

## 2.4 ROS1/ROS2 异构通信架构设计要点

ROS1 与 ROS2 在底层通信栈上存在根本性差异（详见第3章），无法在网络层直接互通消息。本系统设计跨板通信桥接方案时遵循以下原则：

**（1）最小化跨板消息集合**：跨板仅传输巡检业务必需的 `nav_goal` 与 `nav_result` 两类消息，其余传感器、定位与控制闭环话题均在香橙派 ROS1 域内流转，避免跨版本话题透明桥接带来的兼容与时序风险。

**（2）跨板模块与导航侧解耦**：跨板通信模块部署于香橙派3LTS（ROS1 Noetic）侧，与 `move_base` 形成弱耦合接口（目标输入、结果输出）；树莓派5（ROS2 Jazzy）端无需安装任何 ROS1 软件，仅作为任务调度与视觉计算平台。

**（3）时间戳一致性保障**：跨板业务消息（目标/结果）对严格的 TF 时间戳不敏感，但为便于日志对齐与实验测量，两板仍建议通过 NTP/chrony 进行时钟同步（实测时差 < 10ms）。

**（4）频率管控**：跨板消息频率由巡检点数量与到点事件驱动（通常远低于 1Hz），不会对百兆以太网带宽形成压力；导航控制闭环在香橙派本地完成，避免跨板高频小包引起的吞吐波动。

具体跨板通信方案的技术选型论证与部署实现详见第3章。

## 2.5 软件模块划分与工作空间组织

### 2.5.1 香橙派 ROS1 工作空间

香橙派3LTS 运行的 ROS1 Noetic 工作空间（`~/bingda_ros1_noetic-nano/`）包含以下核心功能包：

| 包名 | 功能 | 主要节点/文件 |
|:---|:---|:---|
| `base_control` | 差速底盘串口控制 | `base_control.py`（发布 `/odom`，订阅 `/cmd_vel`） |
| `ydlidar_ros_driver` | EAI 激光雷达 ROS1 驱动 | `ydlidar_ros_driver_node`（发布 `/scan`） |
| `robot_description` | URDF 模型与 TF 发布 | `nanorobot_description.urdf`，`robot_state_publisher` |
| `robot_navigation` | 建图/定位/导航与跨板接口 | `slam_gmapping`、`amcl`、`move_base` 配置与 `zmq_nav_bridge.py` |

`base_control.py` 实现了差速底盘的完整控制逻辑：以10Hz 频率通过串口接收 STM32 上报的编码器增量，计算里程计位姿并发布 `/odom` 话题；同时订阅本机导航框架（`move_base`）发布的 `/cmd_vel` 话题，将线速度/角速度指令转换为左右轮速度并通过串口下发至 STM32。

### 2.5.2 树莓派5 ROS2 工作空间

树莓派5运行的 ROS2 Jazzy 工作空间（以本仓库 `~/smart_robot-main/` 为准，适配 Jazzy）包含以下核心功能包：

| 包名 | 功能 | 主要文件 |
|:---|:---|:---|
| `robot_vision_ros2` | YOLOv8n 检测与巡检调度 | `yolo_detector.py`、`inspection_manager.py`、`waypoint_patrol.py` |

| `robot_navigation_ros2`（扩展） | ROS2 导航/SLAM 备选方案 | Cartographer/Nav2 的参数与启动文件（不作为最终集成主线） |

两个工作空间的主要依赖关系如图2-3所示：香橙派 ROS1 侧各发布节点在本地构成导航闭环，树莓派5 ROS2 侧的巡检调度、视觉检测和记录管理节点在本地干预，两板仅通过 ZeroMQ 传输导航业务消息。

### 2.5.3 系统工作流程

系统完整工作流程分为两个阶段（见图2-4）：

**阶段一：建图阶段（约5~20分钟，一次性操作）**

1. 启动香橙派 ROS1 节点（底盘、雷达、TF），启动 `slam_gmapping` 建图节点；
2. 通过遥控器操控机器人在目标区域行驶一圈，实时构建二维栅格地图；
3. 完成建图后，通过 `map_server`/`map_saver` 将地图保存为 `map.pgm` + `map.yaml`；
4. 关闭建图节点，进入导航阶段。

**阶段二：自主巡检阶段（持续运行）**

1. 启动香橙派 `map_server` + `amcl` + `move_base`，载入保存的地图并进入导航模式；
2. 在 RViz 中手动给定初始位姿（`/initialpose`，精度约 ±30cm 即可）；
3. 启动香橙派跨板 ZMQ 导航接口节点（`zmq_nav_bridge`）；
4. 在树莓派5启动巡检任务调度节点，按预设巡检点顺序下发 `nav_goal` 并等待 `nav_result`；
5. 每次收到到点成功回执后，触发 YOLOv8n 单次推理并记录检测结果；
6. 检测异常时输出告警信息；随后下发下一巡检点目标，循环执行直至完成任务。
