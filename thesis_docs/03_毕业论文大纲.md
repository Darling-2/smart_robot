## 摘要

### 中文摘要

随着人工智能与机器人技术的深度融合，以移动机器人为载体的自主巡检系统在智能建筑与公共安全管理领域具有广泛的应用前景。然而，现有系统普遍面临底层硬件驱动稳定性与上层感知决策先进性难以兼顾、低算力嵌入式平台多任务协同资源受限、目标检测精度与实时性矛盾等挑战。

本文研究并实现了一套面向室内场景的移动机器人自主巡检系统。以 NanoRobot 差速驱动底盘为载体，采用**异构双处理器架构**：**香橙派3LTS运行 ROS1 Noetic**，负责底盘运动控制（`base_control.py` 串口驱动 STM32）与单线激光雷达数据采集，并在本板侧完成 SLAM 建图、定位与自主导航闭环；**树莓派5（4GB）运行 ROS2 Jazzy**，负责 YOLOv8n 视觉检测与巡检任务调度。两板通过百兆以太网直连，采用 **ZeroMQ（ZMQ）业务消息**进行跨板协同，仅传输导航目标与执行结果回执（`nav_goal`/`nav_result`），实现“任务下发—导航执行—到点检测—结果记录/告警”的完整流程。

在地图构建方面，采用 GMapping 算法基于单线激光雷达构建室内二维栅格地图（分辨率 0.05m），并配合 AMCL 实现实时定位。在自主导航方面，基于 ROS1 `move_base` 导航栈实现路径规划与局部避障控制（局部规划器以 DWA 等常用方案为主）。在视觉感知方面，针对室内设施状态检测任务构建自定义数据集（≥600张标注图像），通过迁移学习微调 YOLOv8n 模型，导出 ONNX 格式后在树莓派5上实现约 18 FPS 的实时推理，mAP@0.5 达到 **XX.X%**。最终系统在室内走廊场景中完成多点顺序自主巡检任务，综合导航成功率达 **XX%**，验证了异构 ROS 架构下低成本嵌入式双板平台执行自主巡检任务的工程可行性。（Nav2/Cartographer/RPP 相关内容作为扩展对比与迁移参考保留。）

**关键词**：移动机器人；自主巡检；异构ROS架构；跨板通信；ZeroMQ；move_base；GMapping；YOLOv8

### Abstract

With the deep integration of artificial intelligence and robotics, autonomous mobile robot inspection systems show broad application prospects in smart building management and public safety. However, existing systems commonly face challenges in balancing the stability of low-level hardware drivers with the advancement of high-level perception and decision-making, limited computational resources on embedded platforms for multi-task collaboration, and the contradiction between detection accuracy and real-time performance.

This paper presents the design and implementation of an autonomous mobile robot inspection system for indoor environments. Built upon a NanoRobot differential-drive chassis, the system adopts a **heterogeneous dual-processor architecture**: the **Orange Pi 3LTS runs ROS1 Noetic** for chassis motion control (`base_control.py` driving STM32 via serial) and single-line LiDAR acquisition, and completes SLAM/localization/navigation as a closed loop on-board; the **Raspberry Pi 5 (4GB) runs ROS2 Jazzy** for YOLOv8n-based visual detection and patrol task scheduling. The two boards are connected via 100Mbps Ethernet and coordinate via **ZeroMQ (ZMQ) business messages** by exchanging only navigation goals and execution results (`nav_goal`/`nav_result`).

For mapping, the GMapping algorithm builds indoor 2D occupancy grid maps (0.05m resolution), combined with AMCL for real-time localization. For navigation, the ROS1 `move_base` stack is configured with A* (NavFn) global planning and DWA-style local obstacle avoidance. For visual perception, a custom dataset (≥600 labeled images) for indoor facility state detection is constructed, YOLOv8n is fine-tuned via transfer learning, and the ONNX-exported model achieves approximately 18 FPS on the Raspberry Pi 5 with mAP@0.5 of **XX.X%**. The complete system is validated in indoor corridor scenarios with multi-point sequential patrol tasks, achieving an overall navigation success rate of **XX%**. (Nav2/Cartographer/RPP are kept as optional comparison and migration references.)

**Keywords**: Mobile Robot; Autonomous Inspection; Heterogeneous ROS Architecture; Inter-board Communication; ZeroMQ; move_base; GMapping; YOLOv8

### 第2章 系统总体设计

**预计字数：3500~4000字**

#### 2.1 需求分析

**功能需求**：
1. 自主完成室内场景二维地图构建；
2. 基于已知地图实现多点顺序自主巡检导航；
3. 遭遇静态与动态障碍物时能够安全规避；
4. 在指定巡检点对目标设施状态进行识别；
5. 将巡检结果以时间戳记录，异常状态触发告警。

**性能指标**：
- SLAM建图关键距离误差 ≤ 5%；
- AMCL定位误差 ≤ ±10cm；
- 导航成功率 ≥ 90%（空旷场景）；
- YOLOv8n 检测 mAP@0.5 ≥ 85%；
- 树莓派5推理帧率 ≥ 15 FPS；
- 跨板通信业务消息（`nav_goal`/`nav_result`）往返时延（RTT）≤ 10ms。

**约束条件**：仅使用单线激光雷达与单目摄像头；处理器受限（树莓派5 4GB / 香橙派3LTS 2GB）；运行于室内结构化走廊场景。

#### 2.2 系统总体架构

**三层架构设计**：

```
任务层（树莓派5 / ROS2 Jazzy）
  巡检任务调度 → 下发导航目标（ZMQ）→ 到点触发检测 → 结果记录/报警

导航层（香橙派3LTS / ROS1 Noetic）
  GMapping SLAM建图  |  AMCL 定位  |  move_base导航闭环

                ↔ 以太网直连（ZMQ业务消息：`nav_goal` / `nav_result`）

执行层（香橙派3LTS / ROS1 Noetic）
  base_control.py → STM32 → 差速底盘
  ydlidar_ros_driver → /scan 话题
```

**跨板数据流**（关键通道）：

| 通道 | 类型 | 发布 | 订阅 | 方向 |
|:---|:---|:---|:---|:---:|
| `nav_goal` | ZMQ(JSON) | inspection_manager(ROS2) | zmq_nav_bridge(ROS1) | ROS2→ROS1 |
| `nav_result` | ZMQ(JSON) | zmq_nav_bridge(ROS1) | inspection_manager(ROS2) | ROS1→ROS2 |

说明：ROS 话题（如 `/scan`、`/odom`、`/cmd_vel`、`/tf`）在香橙派ROS1侧完成导航闭环，不做全量透明跨板桥接；树莓派ROS2侧主要处理相机 `/image_raw`、检测结果与任务调度。

#### 2.4 ROS1/ROS2 异构通信架构设计

**核心挑战**：ROS1 使用基于 XML-RPC 的中心化 Master 节点通信；ROS2 使用基于 DDS 的去中心化点对点通信。两者消息序列化格式不同，无法直接互通，若要进行话题级互通需要引入额外的桥接层。

**本系统解决方案**：考虑到部署复杂度与资源受限平台的可维护性，本系统不对 ROS 话题进行全量透明跨板桥接，而是采用“以太网直连 + ZeroMQ（ZMQ）业务消息”的方式实现跨板协同：树莓派5（ROS2）侧仅下发导航目标点（`nav_goal`），香橙派3LTS（ROS1）侧完成导航闭环并回传执行结果（`nav_result`）。具体实现方案详见第3章。

**香橙派 ROS1 工作空间**（`bingda_ros1_noetic-nano`）：

- `robot_description`：URDF 机器人模型（TF发布）

- `lidar_ros2`：`ydlidar_ros2_driver`（仅作备用，实际由 ROS1 端采集）

**预计字数：3000~3500字**

#### 3.1 ROS1 Noetic 与 ROS2 Jazzy 协议差异分析（约600字）

ROS1 与 ROS2 在底层通信机制上存在根本性差异，这是实现跨版本数据交换必须解决的核心问题。

| 对比维度 | ROS1 Noetic | ROS2 Jazzy |
|:---|:---|:---|
| 通信中间件 | XMLRPC 节点发现 + TCPROS/UDPROS 数据传输 | DDS（FastDDS/Cyclone DDS） |
| 节点发现 | 依赖中心化 `rosmaster` 进程 | 去中心化，DDS 参与方自动发现 |
| 消息序列化 | ROS 自定义序列化格式 | CDR（Common Data Representation） |
| 服务质量（QoS） | 不支持，固定语义 | 完整支持（可靠/尽力交付/历史深度/截止期限等） |
| 生命周期管理 | 无 | Managed Node（配置/未激活/激活/终止状态机） |
| 安全机制 | 无 | SROS2（基于 DDS-Security 的认证与加密） |
| Python 版本 | Python 3.8（Ubuntu 20.04） | Python 3.12（Ubuntu 24.04） |

两者根本性的序列化格式差异导致直接话题共享不可行。面向资源受限的双板平台，本系统选择采用跨板**业务消息层**（ZMQ）实现协同：ROS2侧下发目标，ROS1侧执行导航并回传结果，从而避免为大量高频话题引入跨版本桥接的复杂性。

#### 3.2 跨版本通信方案分析与选型（约800字）

现有 ROS1/ROS2 跨版本通信方案主要分为以下几类：

**方案一：`ros1_bridge`（OSRF官方）**
- 原理：同时链接 ROS1 和 ROS2 库，在运行时动态注册两侧话题并转换消息格式；
- 优势：官方支持，消息类型覆盖全面，透明转发；
- 局限：**需要在运行 bridge 的机器上同时安装 ROS1 和 ROS2 环境**，且 ROS1 Noetic（Ubuntu 20.04）与 ROS2 Jazzy（Ubuntu 24.04）glibc 版本差异大，同机共存困难，通常需要借助 Docker 容器化解决。不满足本系统"树莓派5无需安装 ROS1"的要求。

**方案二：`rosbridge_suite`（WebSocket API）**
- 原理：在 ROS1 端运行 `rosbridge_websocket` 服务器（端口9090），对外暴露基于 JSON/BSON 的 WebSocket API；ROS2 端通过标准 WebSocket 客户端库（`roslibpy`、`websockets` 等）连接并订阅/发布话题，无需 ROS1 安装；
- 优势：**ROS2 端完全无需 ROS1 依赖**，部署简单，跨语言支持好；
- 局限：JSON 序列化相比二进制格式延迟略高（约 5~15ms），高频话题（如 `/tf`，200Hz）需做频率限制。

**方案三：自定义 TCP/UDP 转发（Socket Relay）**
- 原理：在 ROS1 端编写节点，将话题数据序列化后通过 TCP/UDP Socket 发送；ROS2 端编写对应的接收节点并反序列化为 ROS2 消息；
- 优势：完全可控，无外部依赖；
- 局限：需自行处理消息序列化、重连、QoS 等，开发工作量较大，可维护性低。

**方案四：Eclipse Zenoh 桥接**
- 原理：在 ROS1 端部署 `zenoh-bridge-ros1` 将话题映射到 Zenoh 数据空间；ROS2 端使用 Zenoh RMW（`rmw_zenoh`）直接接入同一 Zenoh 会话；
- 优势：现代化架构，低延迟，ROS2 端无需 ROS1；
- 局限：生态尚不成熟，配置相对复杂。

**方案五：ZeroMQ（ZMQ）业务消息（本系统采用）**
- 原理：基于以太网在两板之间建立轻量级消息通道，ROS2侧下发导航目标（JSON），ROS1侧回传执行状态与文本信息；
- 优势：跨板传输数据量小、接口稳定、两侧ROS环境完全解耦，便于部署与问题定位；
- 局限：不提供ROS话题级透明互通，需要根据业务定义消息格式。

**本系统选型**：综合部署复杂度、稳定性与资源占用等因素，本文最终采用“以太网直连 + ZMQ业务消息”的跨板协同方案。

#### 4.1 激光SLAM技术概述（约500字）

SLAM问题的数学表达：

$$p(\mathbf{x}_{1:t}, \mathbf{m} \mid \mathbf{z}_{1:t}, \mathbf{u}_{1:t})$$

其中 $\mathbf{x}_{1:t}$ 为机器人轨迹，$\mathbf{m}$ 为地图，$\mathbf{z}_{1:t}$ 为激光观测，$\mathbf{u}_{1:t}$ 为控制输入。

#### 4.3 （扩展对比）Cartographer 2D SLAM 算法原理（约1200字）

**子图构建与局部扫描匹配**：

局部扫描匹配优化问题：

$$\xi^* = \arg\min_{\xi} \sum_{k=1}^{K} \left[1 - M_{\text{smooth}}\left(T_{\xi} \cdot h_k\right)\right]^2$$

其中 $\xi = (t_x, t_y, \theta)$ 为机器人位姿增量，$T_{\xi}$ 为坐标变换矩阵，$h_k$ 为第 $k$ 个激光测量点，$M_{\text{smooth}}$ 为双三次插值的概率占据图。

**回环检测与全局位姿图优化**：

当新子图覆盖区域与历史子图重叠时，通过分支定界（Branch-and-Bound）搜索方法高效寻找约束，并构建位姿约束 $\xi_{ij}$ 加入全局优化。全局优化目标函数：

$$\min_{\Xi^m, \Xi^s} \frac{1}{2}\sum_{ij} \rho\!\left(\mathbf{E}^2\!\left(\xi_i^m, \xi_j^s; \Sigma_{ij}, \xi_{ij}\right)\right)$$

通过 Ceres Solver 求解稀疏非线性最小二乘问题，输出全局一致的位姿图与地图。

**`bingda_lds_2d.lua` 参数配置说明**（扩展对比，结合工作空间实际文件）：

#### 5.4 NavFn 全局路径规划（约500字）

NavFn 规划器基于 Dijkstra/A* 算法在代价地图上传播势场：

$$f(n) = g(n) + h(n)$$

其中 $g(n)$ 为起点到节点 $n$ 的代价（包含地图代价与膨胀代价），$h(n)$ 为启发式距离估计（A* 模式下使用欧式距离）。`NanoRobot.yaml` 配置：`use_astar: true`，`expected_planner_frequency: 5.0 Hz`。

规划路径自然偏向障碍物膨胀代价低的区域（即走廊中央），实现无碰撞全局路径。

#### 5.5 Regulated Pure Pursuit 局部控制器（约1200字）

**纯追踪算法基础**（Pure Pursuit）：

通过前视距离 $L_d$ 在路径上确定目标跟踪点（carrot point），计算所需转向曲率：

$$\kappa = \frac{2 \sin\alpha}{L_d}$$

其中 $\alpha$ 为机器人朝向与前视目标方向的夹角。由曲率推导差速机器人的角速度：$\omega = v \cdot \kappa$。

**RPP 的"Regulated"扩展**（Macenski 等 [9]）：

在基础纯追踪基础上增加三类调节机制：

1. **速度-前视自适应**（`use_velocity_scaled_lookahead_dist: true`）：
   $$L_d = \max\!\left(L_{d,\min},\ \min\!\left(\frac{v}{\text{lookahead\_time}},\ L_{d,\max}\right)\right)$$
   低速时使用较小前视距离提升精度，高速时增大前视距离保持稳定。

2. **曲率感知减速**（`use_regulated_linear_velocity_scaling: true`）：
   当规划路径曲率半径 $R < R_{\min}$ 时，限制线速度 $v \leq v_{\max} \cdot (R / R_{\min})$，防止高速急转造成打滑或偏离路径。

3. **障碍代价感知减速**（`use_cost_regulated_linear_velocity_scaling: true`）：
   靠近高代价障碍物区域时降低速度，增强安全裕度。

**前向碰撞检测**（`use_collision_detection: true`）：
在当前前视距离内沿规划路径预测碰撞，若检测到碰撞则发布停车指令并触发恢复行为，`max_allowed_time_to_collision_up_to_carrot: 1.0s`。

**RPP 关键参数**（来自 NanoRobot.yaml）：

| 参数 | 值 | 说明 |
|:---|:---:|:---|
| `desired_linear_vel` | 0.25 m/s | 期望行进速度 |
| `lookahead_dist` | 0.6 m | 默认前视距离 |
| `min_lookahead_dist` | 0.3 m | 最小前视（近距低速） |

    def __init__(self):
        super().__init__('inspection_manager')
        # 读取巡检点配置
        self.waypoints = self.load_waypoints('waypoints.yaml')
        # Nav2 导航Action客户端
        self.nav_client = ActionClient(self, NavigateToPose, 'navigate_to_pose')
        # 检测服务客户端（到点触发）
        self.detect_client = self.create_client(TriggerDetection, '/trigger_detection')
        # 结果记录
        self.log_writer = CSVLogWriter('inspection_log.csv')

    async def run_inspection(self):
        for wp in self.waypoints:
            # 1. 导航至巡检点
            await self.navigate_to(wp)
            # 2. 触发检测（Service调用，同步等待结果）
            result = await self.detect_client.call_async(TriggerDetection.Request())
            # 3. 记录结果
            self.log_writer.write(wp.id, result)
            # 4. 异常报警
            if result.is_abnormal:
                self.get_logger().warn(f'[ALERT] 巡检点{wp.id}检测到异常: {result.label}')
        # 5. 返回起始点
        await self.navigate_to(self.waypoints[0])

### 第6章 基于 YOLOv8n 的目标状态检测

**预计字数：5000~5500字**

#### 6.1 目标检测技术发展综述（约700字）

- **两阶段检测器**（Faster R-CNN [20]）：精度高，速度慢（<30 FPS on GPU），不适合嵌入式实时应用；
- **单阶段检测器**：SSD → YOLO 系列，速度快；
- **YOLO系列演进**：YOLOv1(2016)[4] → YOLOv3(2018)[5] → YOLOv4(2020)[13] → YOLOv6(2022)[14] → YOLOv7(2022)[15] → YOLOv8(2023)[6]；
- **YOLO综述** [16]：系统梳理各版本关键创新（CSP、PANet、anchor-free、解耦头等）；
- **选用YOLOv8n的理由**：最新稳定版本；anchor-free 设计减少后处理复杂度；YOLOv8n 参数量仅 3.2M，适合树莓派5 18 FPS 的推理需求。

#### 6.2 YOLOv8 网络架构分析（约1000字）

**整体三段结构**：Backbone → Neck（FPN+PAN）→ Head（Decoupled Anchor-Free Head）

**Backbone：改进 CSPDarknet + C2f 模块**

C2f（Cross Stage Partial with 2 convolutions）模块相比 YOLOv5 的 C3 模块，增加了更多残差连接路径，提升了特征复用效率：

```
输入 → 1×1 Conv（分裂）→ [chunk_1, chunk_2...chunk_n] → concat → 1×1 Conv（融合）
                                  ↑── Bottleneck × n ──┘
```

**Neck：PANet 路径聚合**：
自顶向下（语义增强）+ 自底向上（空间增强）的双向特征融合，输出 P3（80×80）、P4（40×40）、P5（20×20）三尺度特征图，兼顾小中大目标检测。

**Head：解耦 Anchor-Free 检测头**：
- **分类分支**（cls）与**回归分支**（reg）分离，各自独立优化；
- 直接预测边界框中心点与宽高，无需预设锚框（anchor-free）；
- 使用 Distribution Focal Loss（DFL）替代传统 IoU Loss，将边界框预测建模为概率分布，提升小目标定位精度。

**YOLOv8 各版本对比**：

| 版本 | 参数量 | COCO mAP | 推理速度(T4 GPU) |
|:---:|:---:|:---:|:---:|
| YOLOv8n | 3.2M | 37.3% | 0.99ms |
| YOLOv8s | 11.2M | 44.9% | 1.20ms |
| YOLOv8m | 25.9M | 50.2% | 3.23ms |

#### 6.4 迁移学习与模型训练（约800字）

**迁移学习策略**：

Pan & Yang [17] 将迁移学习定义为利用源域知识改善目标域学习效果的方法。本文采用**Fine-tuning（微调）**策略：以 COCO 预训练 YOLOv8n 权重为起点，冻结前10层主干（浅层提取通用边缘/纹理特征，无需重新训练）：

$$\min_{\theta_{11:N}} \mathcal{L}(\hat{y}, y; \theta_{1:10}^*, \theta_{11:N})$$

其中 $\theta_{1:10}^*$ 为冻结的预训练参数，仅更新 $\theta_{11:N}$（深层语义特征 + 检测头）。

**训练配置**（`yolo_training/train.py`）：

    epochs=150,

#### 6.6 检测 ROS2 节点封装（约500字）

`yolov8_detector_node.py` 核心设计：

```python
class YOLOv8DetectorNode(Node):
    def __init__(self):
        super().__init__('yolov8_detector')
        # 订阅图像话题
        self.image_sub = self.create_subscription(
            Image, '/image_raw', self.cache_image, 10)
        # 提供检测服务（到点触发）
        self.srv = self.create_service(
            TriggerDetection, '/trigger_detection', self.detect_callback)
        # 加载 ONNX 模型
        self.model = InferenceSession('best.onnx', providers=['CPUExecutionProvider'])
        self.bridge = CvBridge()
        self.latest_image = None

    def cache_image(self, msg):
        """缓存最新帧，等待服务调用时使用"""
        self.latest_image = self.bridge.imgmsg_to_cv2(msg, 'bgr8')

    def detect_callback(self, request, response):
        """被 inspection_manager 调用，执行单次检测"""
        if self.latest_image is None:
            response.success = False
            return response
        detections = self.infer(self.latest_image)
        response.success = True
        response.label = detections[0].label if detections else 'unknown'
        response.confidence = detections[0].confidence if detections else 0.0
        response.is_abnormal = (response.label == 'open')
        return response
```

### 第8章 总结与展望

**预计字数：2000字**

#### 8.1 工作总结

本文围绕移动机器人自主巡检这一核心目标，研究并实现了以下工作：

1. **异构ROS架构设计与跨板通信实现**：提出并实现了 ROS1 Noetic（香橙派3LTS）+ ROS2 Jazzy（树莓派5）的异构双处理器协同方案，采用以太网直连 + ZMQ业务消息实现跨板协同（`nav_goal`/`nav_result`），在不引入ROS话题透明桥接的前提下完成任务下发与结果回传；

2. **GMapping SLAM 建图与 AMCL 精确定位**：在室内走廊场景完成二维栅格建图，关键距离误差 ≤ 5%，AMCL 定位误差 < 5cm；

3. **move_base 自主导航与局部避障控制**：在香橙派ROS1侧配置 `move_base` 导航栈（全局规划+局部规划），实现室内走廊环境下的路径跟踪与障碍物规避，综合导航成功率达 XX%；

4. **YOLOv8n 轻量化目标检测**：构建自定义数据集，通过迁移学习微调 YOLOv8n，ONNX部署后在树莓派5上实现 18 FPS 实时推理，mAP@0.5 达 XX%；

5. **完整闭环巡检系统**：实现"建图—定位—导航—检测—记录—报警"完整闭环，在室内多场景实验中验证了系统可行性。

#### 8.2 不足与展望

**当前不足**：
1. 单线激光雷达维度单一，对非扫描层面障碍物无感知；
2. 检测目标类别少，数据集规模有限；
3. 跨板协同依赖以太网链路稳定性，ZMQ业务消息需进一步完善断链检测、重连与超时回退策略；
4. 巡检结果仅本地日志，未实现远程实时监控。

**未来展望**：
1. **引入深度相机**（如 Intel RealSense D435），实现三维点云障碍感知；
2. **拓展检测目标类别**（仪表盘读数、灭火器状态、配电柜开关状态等）；
3. **探索 micro-ROS 方案**：将 STM32 直接接入 ROS2 生态，彻底取消 ROS1 层，简化架构；
4. **Web端巡检监控界面**：基于 ROS2 Web Bridge 实现实时告警推送；
5. **多机协同巡检**：多机器人任务分配，提升大型场景巡检效率。
