# 第3章 ROS1/ROS2 异构双板通信架构实现

## 3.1 ROS1 Noetic 与 ROS2 Jazzy 协议差异分析

### 3.1.1 ROS1 通信机制

ROS1 的节点通信以**中心化 Master 进程**（`rosmaster`）为核心。各节点在启动时向 `rosmaster` 注册自身提供或需要的话题/服务，`rosmaster` 充当名称服务器，协助发布者与订阅者互相查找对端地址，之后双方建立直连的 **TCPROS**（或 UDPROS）传输链路，后续数据不再经过 `rosmaster` 中转。这一架构虽然简洁，但存在以下固有缺陷：

1. **单点故障**：`rosmaster` 一旦崩溃，整个 ROS1 网络的新连接建立即告失败，已建立连接的通信不受影响，但重启后无法自动恢复；
2. **无实时保障**：TCPROS 建立于 Linux TCP Socket 之上，无法对消息传输时延提供上界保证，不满足安全关键应用的实时性需求；
3. **无安全机制**：任意网段可达的主机均可订阅/发布话题，无认证与加密；
4. **消息序列化格式私有**：ROS1 使用自定义的二进制序列化格式（基于小端字节序的 CDR 变体），不与其他中间件兼容。

### 3.1.2 ROS2 通信机制

ROS2 抛弃了中心化 Master 的设计，转而基于 **DDS（Data Distribution Service）** 标准中间件重构整个通信底层。DDS 由对象管理组织（OMG）标准化，提供以下关键能力：

- **去中心化节点发现（Discovery）**：节点在同一 DDS 域（Domain ID）内通过多播广播自动发现彼此，无需中心注册服务；
- **完整的 QoS 策略**：支持可靠交付（RELIABLE）/ 尽力交付（BEST_EFFORT）、历史深度、截止期限（Deadline）、活跃性（Liveliness）等20+策略，可精确匹配不同话题对传输质量的差异化需求；
- **标准消息序列化**：采用 CDR（Common Data Representation）格式，具备跨语言、跨平台互操作性；
- **生命周期节点**：Managed Node 引入配置/未激活/激活/终止/错误状态机，支持更精细的节点生命周期管理；
- **安全扩展（SROS2）**：基于 DDS-Security 标准提供节点身份认证、消息加密与访问控制。

### 3.1.3 两者互通的根本障碍

如表3-1所示，ROS1 与 ROS2 在通信中间件、节点发现机制和消息序列化格式三个维度均存在根本性差异，导致两者无法在网络层直接互通消息：

| 对比维度 | ROS1 Noetic | ROS2 Jazzy |
|:---|:---|:---|
| 通信中间件 | XMLRPC（节点发现）+ TCPROS（数据传输） | DDS（FastDDS / Cyclone DDS 等） |
| 节点发现 | 中心化 `rosmaster` | DDS 去中心化多播发现 |
| 消息序列化 | ROS1 私有二进制格式 | CDR（Common Data Representation） |
| QoS 策略 | 不支持，固定语义 | 完整 DDS QoS（20+ 策略） |
| 生命周期管理 | 无 | Managed Node 状态机 |
| 安全机制 | 无 | SROS2（DDS-Security） |
| Python 版本 | Python 3.8（Ubuntu 20.04） | Python 3.12（Ubuntu 24.04） |
| Ubuntu 版本 | 20.04 Focal（glibc 2.31） | 24.04 Noble（glibc 2.39） |

**表3-1 ROS1 Noetic 与 ROS2 Jazzy 底层通信机制对比**

尤其是 glibc 版本差异（2.31 vs 2.39），使得在同一 Linux 系统上同时安装 ROS1 Noetic 与 ROS2 Jazzy 会面临严重的动态链接库冲突，通常需要借助容器化方案才能并行运行。因此，**本系统明确要求跨板通信方案不依赖树莓派5（ROS2 Jazzy）上安装 ROS1 环境**，跨板接口模块部署于香橙派3LTS（ROS1 Noetic）侧，树莓派5通过以太网对接该接口服务。

## 3.2 跨版本通信方案分析与选型

### 3.2.1 方案一：`ros1_bridge`（OSRF 官方方案）

**原理**：`ros1_bridge` 是 OSRF 专为 ROS1/ROS2 互通开发的官方桥接软件包。它同时链接 ROS1 和 ROS2 的客户端库，在运行时动态注册两侧已知话题，并在消息到达时进行 ROS1 私有格式 ↔ CDR 格式的转换，实现话题的双向透明代理转发。

**优势**：官方支持、消息类型覆盖全面（标准消息类型自动映射，自定义消息类型可手动注册）、转发延迟低（纯内存格式转换，不涉及网络序列化开销）。

**关键局限**：`ros1_bridge` 的运行时同时依赖 ROS1 和 ROS2 的共享库，要求**运行 bridge 的主机上必须同时安装 ROS1 和 ROS2 环境**。在本系统的部署约束下：
- 若将 bridge 运行于香橙派（Ubuntu 20.04），需在其上额外安装 ROS2 Jazzy，但 Jazzy 要求 Ubuntu 24.04，在 20.04 上无官方支持，需要从源码编译或使用非官方 apt 源，显著增加部署风险；
- 若将 bridge 运行于树莓派5（Ubuntu 24.04），需在其上额外安装 ROS1 Noetic，同样面临 Ubuntu 版本不匹配问题，通常借助 Docker 解决，但引入了额外的容器化复杂度与启动时延。

综合评估，`ros1_bridge` **不满足"树莓派5无需 ROS1 依赖"的约束**，不作为首选方案。

### 3.2.2 方案二：`rosbridge_suite`（WebSocket 协议）

**原理**：`rosbridge_suite` 在 ROS1 端部署 `rosbridge_websocket` 服务器（默认端口 9090），对外暴露基于 **JSON 或 BSON** 编码的 WebSocket API（ROSBridge Protocol v2.0）；ROS2 端通过标准 WebSocket 客户端库（Python 的 `roslibpy` 或 `websockets`，JavaScript 的 `roslibjs` 等）连接此服务，无需安装任何 ROS1 软件即可订阅/发布话题、调用服务。

**通信流程**：
```
[树莓派5 ROS2 节点]
    ↕  ros2 topic pub/sub（本地 DDS）
[ROS2 WebSocket 客户端节点]
    ↕  WebSocket（JSON，以太网 TCP/IP）
[香橙派 rosbridge_websocket 服务器]（ROS1）
    ↕  ROS1 TCPROS
[香橙派 ROS1 节点]（/scan、/odom、/cmd_vel、/tf 等）
```

**优势**：
- ROS2 端**完全无需 ROS1 依赖**，树莓派5仅需标准 Python WebSocket 库；
- 部署极简：香橙派端仅需 `sudo apt install ros-noetic-rosbridge-suite` 并执行一条 `roslaunch` 命令；
- 跨语言、跨平台兼容性好，已有成熟的 Python/JavaScript/C# 客户端实现；
- 可动态订阅/取消订阅话题，灵活性高。

**局限**：
- JSON 文本序列化相比 CDR 二进制格式有额外开销，单次消息延迟约 5~15ms（视消息大小而定）；
- 高频大消息（如 `/scan` 10Hz、每帧约 3KB）在 JSON 模式下带宽消耗约为二进制的2~3倍，但百兆以太网链路（有效吞吐约 80Mbps）对本系统话题集合（合计 < 0.5Mbps）仍绰绰有余；
- 对 `/tf` 等高频话题需配置频率限制，避免大量小包冲击 WebSocket 服务器。

### 3.2.3 方案三：自定义 TCP/UDP Socket 转发

**原理**：在 ROS1 端编写专用转发节点，订阅指定话题后将 ROS1 消息序列化（通过 `rospy` 的消息序列化接口）并通过 TCP/UDP Socket 发送至树莓派5；树莓派5端编写对应接收节点，将收到的字节流反序列化为 ROS2 消息并重新发布。

**评估**：该方案完全可控，无外部依赖。但需要手动处理消息类型注册、重连机制、粘包/分包处理、时间戳对齐等细节，开发与维护成本较高，且对每种消息类型均需编写专用代码，扩展性差。对于本系统原型验证阶段，性价比不高。

### 3.2.4 方案四：Eclipse Zenoh 桥接

**原理**：Eclipse Zenoh 是一种基于"数据空间"（Data Space）概念的新一代分布式通信框架，定位为 DDS 的现代化替代。`zenoh-bridge-ros1` 插件可将 ROS1 话题映射至 Zenoh 路由，ROS2 端可通过 `rmw_zenoh`（ROS2 的 Zenoh RMW 实现）直接接入同一 Zenoh 会话，实现零 ROS1 依赖的跨版本通信，且延迟极低。

**评估**：Zenoh 架构先进，是未来机器人通信生态的有力方向，Macenski 等人 [2] 的综述也对其前景给予肯定。但截至本系统开发时，`zenoh-bridge-ros1` 的稳定性与社区文档相对有限，在嵌入式 ARM 平台上的兼容性存在不确定性，暂不作为生产环境首选。

### 3.2.5 方案五：ZeroMQ（ZMQ）应用层业务消息（最终采用）

**原理**：在 ROS1 端编写一个轻量跨板接口节点 `zmq_nav_bridge`，使用 ZeroMQ 提供的 PUSH/PULL 与 PUB/SUB 通信模式，在以太网链路上传输 **JSON 格式的业务消息**。树莓派侧仅需 Python `pyzmq` 依赖，即可向香橙派侧下发导航目标（`nav_goal`），并订阅导航结果回执（`nav_result`）。

**通信流程**：
```
[树莓派5 ROS2 巡检调度]
  │  ZMQ PUSH（nav_goal，JSON）
  ▼
[香橙派3LTS zmq_nav_bridge]
  │  发布 /move_base_simple/goal
  ▼
[香橙派3LTS move_base 导航栈]
  │  /move_base/result
  ▼
[香橙派3LTS zmq_nav_bridge]
  │  ZMQ PUB（nav_result，JSON）
  ▼
[树莓派5 ROS2 巡检调度]
```

**优势**：
- 树莓派侧无需 ROS1，且无需 ROS Topic 级透明桥接，部署边界清晰；
- 跨板仅传输目标与结果，消息频率低、带宽占用极小；
- ZeroMQ 自带重连能力与多种通信模式，工程实现简洁。

**局限**：
- 非 ROS 原生话题互通，无法“透明订阅/发布任意话题”，适用前提是跨板交互的语义边界清晰（本系统为“导航目标/结果”）。

### 3.2.6 方案综合对比与选型

## 3.3 本系统通信方案部署实现

### 3.3.1 网络基础配置

双板通过一根 CAT5e 百兆以太网网线直连，配置静态 IP（避免 DHCP 协商延迟）：

```bash
# 香橙派3LTS（/etc/netplan/01-netcfg.yaml）
network:
  version: 2
  renderer: networkd
  ethernets:
    eth0:
      addresses: [192.168.50.2/24]
      dhcp4: false

# 树莓派5（/etc/netplan/01-netcfg.yaml）
network:
  version: 2
  renderer: networkd
  ethernets:
    eth0:
      addresses: [192.168.50.3/24]
      dhcp4: false
```

配置完成后执行 `sudo netplan apply` 使配置生效。验证连通性：

```bash
# 在树莓派5上执行
ping -c 10 192.168.50.2
# 预期输出：rtt min/avg/max = 0.3/0.5/1.2 ms
```

同时，在两板的 `/etc/hosts` 中分别添加对端主机名映射，便于 ROS 话题路由：

```
# 香橙派 /etc/hosts 追加：
192.168.50.3  rpi5
# 树莓派5 /etc/hosts 追加：
192.168.50.2  orangepi
```

### 3.3.2 时间同步配置

ROS 的 TF 变换查找（`tf2::Buffer::lookupTransform`）对时间戳一致性敏感，两板时钟偏差过大会导致 TF 查询超时（`"Lookup would require extrapolation into the future"`）。本系统通过 `chrony` 进行网络时间同步：

```bash
# 香橙派端作为本地 NTP 服务器（无公网时可离线同步）
# /etc/chrony/chrony.conf 追加：
allow 192.168.50.0/24
local stratum 8

# 树莓派5端同步至香橙派
# /etc/chrony/chrony.conf 追加：
server 192.168.50.2 iburst prefer
```

执行 `sudo systemctl restart chrony` 后，通过 `chronyc tracking` 验证时钟偏差（System time offset）应 < 10ms。跨板业务消息（目标/结果）的延迟测量与日志对齐可利用该同步结果。

### 3.3.3 跨板 ZMQ 接口配置（`nav_goal/nav_result`）

本系统跨板通信采用 ZeroMQ（ZMQ）传输业务级 JSON 消息。香橙派侧的 `zmq_nav_bridge` 负责将 `nav_goal` 写入 ROS1 导航接口（发布 `/move_base_simple/goal`），并将 `move_base` 的执行结果（订阅 `/move_base/result`）封装为 `nav_result` 回传树莓派。

**端口与连接方式**：

- 香橙派（服务端）
  - PULL 绑定：`tcp://0.0.0.0:5555`（接收 `nav_goal`）
  - PUB 绑定：`tcp://0.0.0.0:5556`（发布 `nav_result`）
- 树莓派（客户端）
  - PUSH 连接：`tcp://192.168.50.2:5555`
  - SUB 连接：`tcp://192.168.50.2:5556`

**香橙派端节点启动**（示例）：

```bash
source /opt/ros/noetic/setup.bash
source ~/bingda_ros1_noetic-nano/devel/setup.bash
roslaunch robot_navigation zmq_bridge.launch
```

**树莓派5端客户端参数**：

树莓派侧巡检调度节点在配置中指定香橙派 IP 与端口（如 `192.168.50.2:5555/5556`），启动后按巡检点序列发送 `nav_goal` 并等待对应 `goal_id` 的 `nav_result` 回执。

**消息格式（JSON）**：

| 字段 | 类型 | 说明 |
|:---|:---:|:---|
| `type` | string | `nav_goal` 或 `nav_result` |
| `goal_id` | string | 目标唯一标识，用于结果匹配 |
| `x,y,yaw` | float | 目标位姿（平面坐标 + 偏航角） |
| `frame_id` | string | 目标坐标系（通常为 `map`） |
| `status` | int | 结果状态码（成功/失败/取消等，按实现映射） |
| `text` | string | 结果文本描述（可选） |
| `timestamp` | float/int | 发送时间戳（可选，用于测延迟） |

### 3.3.4 ROS 网络环境变量说明

ZMQ 业务消息方案不依赖 ROS1/ROS2 跨板网络发现机制，树莓派侧无需配置 `ROS_MASTER_URI` 等 ROS1 环境变量。香橙派侧 ROS1 导航栈默认使用本机 `roscore`（`ROS_MASTER_URI` 默认为 `http://localhost:11311`）即可。

如需在树莓派或其他主机上远程运行 RViz（ROS1）观测香橙派导航状态，可按 ROS1 常规网络配置设置：

```bash
export ROS_MASTER_URI=http://192.168.50.2:11311
export ROS_IP=192.168.50.2
```

## 3.4 关键接口配置与验证

### 3.4.1 TF 坐标变换树配置（ROS1 本地）

ROS 导航栈的正常运行依赖完整且连续的 TF 坐标变换树，本系统涉及的 TF 链路如下：

```
map
 └─ odom           （由 AMCL 维护，香橙派 ROS1）
   └─ base_link  （由 base_control.py 维护，香橙派 ROS1）
     └─ laser_link  （由 robot_state_publisher 维护，香橙派 ROS1，静态变换）
```

其中：
- `map → odom`：AMCL 输出的定位修正变换；
- `odom → base_link`：底盘里程计变换，由香橙派 `base_control.py` 发布；
- `base_link → laser_link`：激光雷达安装偏置（静态），由香橙派 `robot_state_publisher` 发布。

由于导航闭环均在香橙派 ROS1 域内完成，TF 链路无需跨板同步；树莓派侧仅通过 `nav_goal/nav_result` 与导航侧进行弱耦合交互。

### 3.4.2 接口验证步骤

系统启动后，按以下步骤验证导航侧与跨板接口状态：

**步骤1：验证香橙派 ROS1 侧导航闭环话题**
```bash
rostopic hz /scan     # 预期：约 10 Hz
rostopic echo /odom --once
rostopic echo /cmd_vel --once
```

**步骤2：验证 ZMQ 端口连通性**

在树莓派侧对香橙派进行端口连通性检查（示例）：
```bash
ping -c 5 192.168.50.2
```

**步骤3：验证 nav_goal/nav_result 业务闭环**

启动树莓派巡检调度节点，发送一个测试目标点（可选取近距离安全点）。观察香橙派侧 `move_base` 是否收到目标并开始规划，同时树莓派侧是否在导航结束后收到 `nav_result` 回执。

## 3.5 通信性能测试

### 3.5.1 端对端延迟测量方法

### 3.5.3 对自主巡检任务节拍的影响分析

由于底盘控制闭环与局部避障均在香橙派 ROS1 侧完成，跨板通信延迟不会进入 `/cmd_vel` 控制回路。跨板 ZMQ 消息的影响主要体现在“目标下发 → 收到回执”的任务节拍上。

设单个巡检点的平均导航耗时为 $T_{nav}$，跨板消息往返延迟为 $d$（RTT），则单点任务的通信额外开销近似为 $d$，相对开销比例为：

$$\eta = \frac{d}{T_{nav}} \times 100\%$$

当 $T_{nav}$ 通常为数十秒量级时，即使 $d$ 为数毫秒，其占比也远小于 1%，对整体巡检周期影响可忽略。

### 3.5.4 带宽占用分析

系统运行时，跨板仅传输 `nav_goal/nav_result` 两类消息，带宽消耗估算如下：

| 消息 | 消息大小 | 频率（典型） | 单向带宽 |
|:---:|:---:|:---:|:---:|
| `nav_goal` | ~0.1 KB | 低频（到点触发） | 约可忽略 |
| `nav_result` | ~0.1 KB | 低频（到点触发） | 约可忽略 |
| **合计** | | | **远低于 1 KB/s** |

百兆以太网链路理论带宽为 100Mbps（实际有效吞吐约 80Mbps），本系统跨板业务消息带宽需求几乎可忽略，具有充裕余量。

## 3.6 本章小结

本章深入分析了 ROS1 Noetic 与 ROS2 Jazzy 在通信中间件、节点发现机制和消息序列化格式方面的根本性差异，论证了两者直接互通的不可行性。在此基础上，系统性地对比评估了 `ros1_bridge`、`rosbridge_suite`、自定义 Socket 转发、Eclipse Zenoh 以及 ZeroMQ 业务级消息等跨版本通信思路，从部署约束（树莓派5无需 ROS1）、工程成熟度和嵌入式平台兼容性等维度进行综合权衡，确定了本系统最终采用的跨板通信方法。详细阐述了网络基础配置（静态 IP、时钟同步）与 ZMQ 接口部署，并给出了接口验证步骤。性能测试与理论分析表明，跨板消息延迟处于毫秒级且带宽占用极低，能够满足自主巡检任务的调度需求，为系统集成与综合实验奠定了通信基础。
