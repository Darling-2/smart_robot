# 第5章 基于move_base的自主导航与障碍规避

## 5.1 move_base导航框架概述

### 5.1.1 框架设计思想与架构

`move_base` 是 ROS1 官方提供的移动机器人导航框架，集成了全局路径规划、局部路径跟踪与动态障碍回避、代价地图维护和故障恢复等核心功能 [7]。在本系统中，`move_base` 运行于香橙派3LTS（ROS1 Noetic）上，与底盘控制节点（`base_control`）、激光雷达驱动（`ydlidar_ros_driver`）和 AMCL 定位节点共同构成完整的导航闭环。树莓派5侧的巡检调度节点（`waypoint_patrol`）通过第3章所述的 ZMQ 跨板桥接机制将导航目标发送至香橙派端的 `zmq_nav_bridge` 节点，由其转发为 `/move_base_simple/goal` 话题触发导航任务。

`move_base` 采用**全局规划+局部控制**的双层规划架构。全局规划器（Global Planner）在已知地图的全局代价地图上搜索从当前位置到目标点的最优路径；局部规划器（Local Planner）在实时感知范围内，沿全局路径方向搜索既安全又高效的速度指令，并以 `/cmd_vel` 话题发布给底盘控制器。当路径被阻塞或规划失败时，`move_base` 自动触发恢复行为（代价地图清除、等待等），尝试解除困境后重新规划。

**导航闭环在香橙派本地完成**是本系统的重要设计决策。`/scan`（激光雷达数据）→ 代价地图 → 路径规划 → `/cmd_vel`（速度指令）的整条数据链路均在 ROS1 本地话题总线上传输，不经过跨板网络，从而彻底消除了实时控制环路的网络延迟与丢包风险。树莓派5仅通过 ZMQ 发送目标坐标并接收导航结果（非实时数据），对网络延迟不敏感。

### 5.1.3 启动配置

本系统导航栈通过 `robot_navigation.launch` 统一启动，其中整合了地图加载、AMCL 定位和 `move_base` 路径规划三个核心环节。launch 文件支持通过 `planner` 参数在 DWA 与 TEB 两种局部规划器之间切换（默认 TEB），并通过环境变量 `BASE_TYPE=NanoRobot` 加载对应的参数文件集：

```xml
<!-- navigation_stack.launch 核心结构 -->
<node pkg="map_server" name="map_server" type="map_server" args="$(arg map_file)">
  <param name="frame_id" value="map"/>
</node>
<node pkg="amcl" type="amcl" name="amcl" output="screen">
  <rosparam file="$(find robot_navigation)/param/$(env BASE_TYPE)/amcl_params.yaml" command="load"/>
</node>
<include file="$(find robot_navigation)/launch/move_base.launch">
  <arg name="planner" value="$(arg planner)"/>  <!-- dwa 或 teb -->
</include>
```

`move_base.launch` 根据 `planner` 参数动态加载不同的局部规划器配置文件，全局规划器统一使用 `global_planner/GlobalPlanner` 替代默认的 `navfn`，支持 Dijkstra 和 A* 两种搜索算法。

## 5.2 代价地图（Costmap2D）设计与配置

### 5.2.1 代价地图原理

代价地图（Costmap2D）是 `move_base` 中环境表示的核心数据结构，以二维栅格形式存储每个格子的"导航代价"值（0~253，254=致命障碍物，255=未知区域）。代价值越高，规划路径越倾向于绕开该区域，从而在路径代价最小化中自然实现障碍物回避。代价地图采用分层（Layer）架构，各层独立更新后叠加生成最终代价地图：

- **静态层（StaticLayer）**：订阅 `/map` 话题，加载 AMCL 当前使用的已知地图（`map.pgm`），将地图中的障碍物格子标记为最大代价（254），自由区域标记为 0；
- **障碍层（ObstacleLayer/VoxelLayer）**：实时订阅 `/scan` 话题，将激光探测到的障碍物标记为最大代价，并通过**射线追踪（Ray Casting）**将激光路径上的格子清零（标记为可通行），动态清除已消失的临时障碍物；
- **膨胀层（InflationLayer）**：以每个障碍物格子为中心，在半径 $r$ 内按指数衰减函数赋予周围格子代价，形成"安全缓冲区"。膨胀代价计算公式为：

$$\text{cost}(d) = \text{INSCRIBED\_COST} \cdot e^{-\lambda \cdot (d - r_{\text{inscribed}})}$$

其中 $d$ 为到最近障碍物的距离，$\lambda$ 为代价衰减因子（`cost_scaling_factor`），$r_{\text{inscribed}}$ 为机器人内切圆半径（约 0.10m），确保机器人在任何方向均不与障碍物碰撞。

### 5.2.2 公共代价地图参数

全局与局部代价地图共享 `costmap_common_params.yaml` 中的公共参数，包括机器人轮廓、障碍物感知和膨胀配置：

```yaml
# costmap_common_params.yaml（NanoRobot）
footprint: [[-0.09, -0.1], [0.04, -0.1], [0.04, 0.1], [-0.09, 0.1]]

obstacle_layer:
  enabled: true
  obstacle_range: 2.5          # 障碍物标记最大范围 2.5m
  raytrace_range: 3.0          # 射线追踪清除范围 3.0m
  track_unknown_space: false
  combination_method: 1
  observation_sources: laser_scan_sensor
  laser_scan_sensor:
    data_type: LaserScan
    topic: scan
    marking: true              # 将激光检测到的障碍物标记在代价地图中
    clearing: true             # 通过射线追踪清除消失的障碍物

inflation_layer:
  enabled: true
  cost_scaling_factor: 10.0    # 代价衰减速率（值越大衰减越快，安全距离越小）
  inflation_radius: 0.1        # 膨胀半径 0.1m
```

**机器人轮廓说明**：机器人矩形轮廓为 `[[-0.09,-0.1],[0.04,-0.1],[0.04,0.1],[-0.09,0.1]]`（单位 m），长约 0.13m、宽 0.20m，外切圆半径约 0.13m。

**膨胀参数设计**：`inflation_radius: 0.1m` 仅略大于机器人内切圆半径，`cost_scaling_factor: 10.0` 使代价衰减速率较快。这一"紧凑膨胀"策略的设计考量是：本系统工作在宽度约 1.0~1.5m 的室内走廊中，若膨胀半径过大（如 0.3m 以上），走廊两侧墙壁的膨胀区域将大面积重叠，导致 GlobalPlanner 在窄走廊中频繁规划失败。紧凑膨胀使全局路径能够在窄通道中正常通过，由局部规划器（DWA/TEB）的避障机制保证实际运行时的安全距离。

### 5.2.3 全局代价地图配置

全局代价地图覆盖整个已知地图范围，用于 GlobalPlanner 全局路径规划：

```yaml
# global_costmap_params.yaml
global_costmap:
  global_frame: map
  robot_base_frame: base_footprint
  update_frequency: 1.0          # 更新频率 1Hz（全图更新代价较高，降频合理）
  publish_frequency: 0.5         # 发布频率 0.5Hz
  transform_tolerance: 0.5
  plugins:
    - {name: static_layer,     type: "costmap_2d::StaticLayer"}
    - {name: obstacle_layer,   type: "costmap_2d::VoxelLayer"}
    - {name: inflation_layer,  type: "costmap_2d::InflationLayer"}
```

全局代价地图使用 `VoxelLayer`（三维体素层）处理障碍物，相比普通 `ObstacleLayer` 多了垂直方向的体素管理，能够在低矮家具环境中更准确地区分地面反射和真实障碍。`robot_base_frame: base_footprint` 与 AMCL 定位和底盘控制中的坐标系名称保持一致。

### 5.2.4 局部代价地图配置

局部代价地图以机器人当前位置为中心的滚动窗口方式运行，以更高频率更新，用于局部规划器实时感知和规避近距障碍物：

```yaml
# local_costmap_params.yaml
local_costmap:
  global_frame: map
  robot_base_frame: base_footprint
  update_frequency: 5.0          # 更新频率 5Hz（更快响应动态障碍）
  publish_frequency: 5.0
  rolling_window: true
  width: 3                       # 滚动窗口 3m×3m
  height: 3
  resolution: 0.05               # 分辨率 5cm，与全局地图一致
  transform_tolerance: 0.5
  plugins:
    - {name: static_layer,     type: "costmap_2d::StaticLayer"}
    - {name: obstacle_layer,   type: "costmap_2d::ObstacleLayer"}
```

局部代价地图更新频率（5Hz）高于全局（1Hz），确保对动态障碍物（如行人经过）的感知延迟 ≤ 200ms。3m×3m 的滚动窗口覆盖了激光雷达 2.5m 障碍标记范围内的全部区域，使局部规划器能够完整利用激光观测信息进行避障决策。局部代价地图不包含膨胀层，由局部规划器自身的避障距离参数保证安全。

## 5.3 GlobalPlanner 全局路径规划

### 5.3.1 算法原理

本系统全局路径规划采用 ROS1 `global_planner` 包提供的 `GlobalPlanner` 插件，替代默认的 `navfn`，支持 **Dijkstra** 和 **A*** 两种搜索算法。系统默认使用 Dijkstra 模式（`use_dijkstra: true`），在全局代价地图上搜索从当前位置到目标点的最短加权路径。

Dijkstra 算法以广度优先方式遍历代价栅格，维护每个格子到起点的累积最小代价 $g(n)$：

$$g(n) = \min_{m \in \text{neighbors}} \left[ g(m) + c(m, n) \right]$$

其中 $c(m, n)$ 为从格子 $m$ 到相邻格子 $n$ 的移动代价，由步长代价（直行 1.0、对角线 $\sqrt{2} \approx 1.414$）与该格子的膨胀代价加权组成。代价缩放因子 `cost_factor: 3` 控制膨胀代价对路径选择的影响程度——值越大，路径越倾向远离障碍物的走廊中央区域。

算法从起点开始逐步扩展至所有可达格子，直到到达目标格子或其容差范围（`default_tolerance: 0.2m`）内的任一格子，然后从目标回溯至起点生成全局路径点序列。

当切换为 A* 模式（`use_dijkstra: false`）时，代价评估函数增加启发项：

$$f(n) = g(n) + h(n)$$

其中 $h(n) = \sqrt{(x_n - x_g)^2 + (y_n - y_g)^2}$ 为欧式距离启发式。A* 通过启发函数加速搜索方向性，在大地图上规划速度显著优于 Dijkstra，但不保证路径相对于代价地图的全局最优性（仅在一致性启发下保证到达性最优）。本系统使用 Dijkstra 保证全局最优路径，其在约 30m×20m 室内地图上的规划耗时约 20~50ms，完全满足 `planner_frequency: 0.2Hz`（每 5 秒重规划一次）的时间要求。

### 5.3.2 关键参数配置

```yaml
# global_planner_params.yaml
GlobalPlanner:
  allow_unknown: true           # 允许穿越地图未知区域规划路径
  default_tolerance: 0.2        # 目标点位置容差 0.2m
  use_grid_path: false          # 使用梯度路径（非网格对齐），弯道更平滑
  use_quadratic: false          # 不使用二次近似势场
  lethal_cost: 253              # 致命代价阈值
  neutral_cost: 50              # 中性代价值（自由空间基础代价）
  cost_factor: 3                # 代价缩放因子
  publish_potential: true       # 发布势场用于 RViz 调试可视化
```

**关键参数说明**：

- `use_grid_path: false` 使 GlobalPlanner 输出**梯度插值路径**而非严格沿栅格对齐的阶梯路径。梯度路径在弯道处产生更平滑的路径点分布，减少局部规划器的跟踪振荡，对差速驱动机器人的轨迹平滑性尤为重要；
- `default_tolerance: 0.2m` 允许目标点附近 20cm 范围内存在障碍物时仍能规划成功，在家具密集的室内场景中显著提高了规划成功率；
- `publish_potential: true` 发布势场话题（`/move_base/GlobalPlanner/potential`），可在 RViz 中以热力图形式可视化代价分布，便于调试和论文截图。

## 5.4 局部路径规划器

本系统通过 `move_base.launch` 中的 `planner` 参数支持 DWA 和 TEB 两种局部规划器的运行时切换。本节重点阐述 DWA 的算法原理与参数配置，并与 TEB 进行系统性对比分析。

### 5.4.1 动态窗口法（DWA）原理

动态窗口法（Dynamic Window Approach, DWA）[10] 是经典的局部避障与路径跟踪算法，其核心思想是在机器人当前速度空间中，考虑运动学约束和动力学极限，搜索最优的线速度-角速度组合 $(v, \omega)$，使机器人在避开障碍物的同时沿全局路径前进。

DWA 的工作流程分为三步：

**第一步：确定动态窗口（Dynamic Window）**

根据机器人当前速度 $(v_c, \omega_c)$ 和加速度极限，确定下一控制周期 $\Delta t$ 内可达的速度范围：

$$V_d = \{(v, \omega) \mid v \in [v_c - \dot{v}_{\max} \cdot \Delta t,\ v_c + \dot{v}_{\max} \cdot \Delta t],\ \omega \in [\omega_c - \dot{\omega}_{\max} \cdot \Delta t,\ \omega_c + \dot{\omega}_{\max} \cdot \Delta t]\}$$

其中 $\dot{v}_{\max} = 2.5\ \text{m/s}^2$（`acc_lim_x`）、$\dot{\omega}_{\max} = 3.2\ \text{rad/s}^2$（`acc_lim_theta`）。动态窗口与运动学硬约束取交集后，得到实际可行速度集合：

$$V_r = V_d \cap \{(v, \omega) \mid v \in [-0.25,\ 0.25]\ \text{m/s},\ \omega \in [-1.0,\ 1.0]\ \text{rad/s}\}$$

**第二步：前向仿真与轨迹生成**

对 $V_r$ 中的候选速度 $(v_i, \omega_i)$ 进行**前向仿真**（`sim_time: 2.0s`），以差速运动学模型模拟机器人在该速度下未来 2 秒的运动轨迹。本系统在线速度方向采样 20 个候选值（`vx_samples: 20`），角速度方向采样 40 个候选值（`vtheta_samples: 40`），共产生约 800 条候选轨迹。

**第三步：多目标评价函数**

对每条候选轨迹 $\tau_i$ 计算综合评分：

$$G(\tau_i) = \alpha \cdot \text{path\_dist}(\tau_i) + \beta \cdot \text{goal\_dist}(\tau_i) + \gamma \cdot \text{occ\_dist}(\tau_i)$$

其中各评价分量的含义与本系统配置权重如下：

| 评价分量 | 含义 | 权重参数 | 本系统值 |
|:---|:---|:---|:---:|
| $\text{path\_dist}(\tau_i)$ | 轨迹末端到全局路径最近点的距离代价 | `path_distance_bias` | $\alpha = 32.0$ |
| $\text{goal\_dist}(\tau_i)$ | 轨迹末端到目标点的距离代价 | `goal_distance_bias` | $\beta = 20.0$ |
| $\text{occ\_dist}(\tau_i)$ | 轨迹上最近障碍物距离的倒数代价 | `occdist_scale` | $\gamma = 0.02$ |

选择评分最优的候选轨迹对应的 $(v^*, \omega^*)$ 作为输出速度指令，发布至 `/cmd_vel` 话题。`forward_point_distance: 0.325m` 在机器人前方 32.5cm 处设置一个额外评估点，引导机器人朝向全局路径方向行驶。

### 5.4.2 DWA 关键参数配置

来自 `dwa_local_planner_params.yaml` 的完整参数：

```yaml
DWAPlannerROS:
  # ---- 机器人速度限制 ----
  max_vel_x: 0.25              # 最大前进速度 0.25 m/s
  min_vel_x: -0.25             # 最大后退速度 0.25 m/s
  max_vel_y: 0.0               # 差速驱动无横向速度
  max_vel_trans: 0.26           # 最大平移合速度 0.26 m/s
  min_vel_trans: 0.13           # 最小平移速度 0.13 m/s
  max_vel_theta: 1.0            # 最大角速度 1.0 rad/s
  min_vel_theta: 0.5            # 最小角速度 0.5 rad/s

  # ---- 加速度限制 ----
  acc_lim_x: 2.5               # 线加速度极限 2.5 m/s²
  acc_lim_y: 0.0               # 差速驱动无横向加速度
  acc_lim_theta: 3.2            # 角加速度极限 3.2 rad/s²

  # ---- 目标容差 ----
  xy_goal_tolerance: 0.1        # 位置容差 10cm
  yaw_goal_tolerance: 0.1       # 航向容差 ~5.7°

  # ---- 前向仿真 ----
  sim_time: 2.0                 # 仿真时间窗口 2.0s
  vx_samples: 20                # 线速度采样数
  vy_samples: 0                 # 差速驱动不采样横向速度
  vtheta_samples: 40            # 角速度采样数

  # ---- 轨迹评分权重 ----
  path_distance_bias: 32.0      # 路径跟踪权重
  goal_distance_bias: 20.0      # 目标趋近权重
  occdist_scale: 0.02           # 障碍回避权重
  forward_point_distance: 0.325 # 前向评估点距离 0.325m
```

**参数调优说明**：

- `max_vel_x: 0.25 m/s` 与底盘控制器（`base_control`）的速度限制一致，确保速度指令不超过电机驱动能力；
- `min_vel_trans: 0.13 m/s` 设置了最小平移速度下限，防止 DWA 在低速时输出过小的速度导致电机死区无法驱动；
- `sim_time: 2.0s` 的仿真窗口在 0.25m/s 速度下覆盖约 0.5m 前方距离，足以在走廊中提前感知交叉口转弯；
- `path_distance_bias: 32.0` 远大于 `occdist_scale: 0.02`，使 DWA 优先保持路径跟踪精度，仅在障碍物非常接近时才大幅偏离全局路径——这符合室内走廊场景中"沿走廊中央线行驶"的期望行为。

### 5.4.3 TEB 局部规划器与对比分析

TEB（Timed Elastic Band）规划器 [11] 将局部路径建模为一系列带时间戳的位姿节点（弹性带），通过求解**非线性最小二乘优化问题**，在满足运动学、动力学和障碍物约束的前提下生成时间最优轨迹。其优化目标函数为：

$$\min_{\mathbf{T}} \sum_{k} \left[ w_t \cdot \Delta t_k^2 + w_{\text{obs}} \cdot c_{\text{obs}}(q_k) + w_v \cdot c_v(q_k, q_{k+1}) + w_\omega \cdot c_\omega(q_k, q_{k+1}) \right]$$

其中 $\mathbf{T} = \{q_1, \Delta t_1, q_2, \Delta t_2, \ldots\}$ 为弹性带节点序列，$w_t$、$w_{\text{obs}}$、$w_v$、$w_\omega$ 分别为时间最优、障碍避让、速度约束和角速度约束的优化权重。

本系统 TEB 局部规划器的关键参数（来自 `teb_local_planner_params.yaml`）：

```yaml
TebLocalPlannerROS:
  # ---- 弹性带参数 ----
  teb_autosize: true            # 自动调整弹性带节点数
  dt_ref: 0.3                   # 节点时间间隔 0.3s
  global_plan_overwrite_orientation: true

  # ---- 机器人约束 ----
  max_vel_x: 0.2                # 最大线速度 0.2 m/s（略低于 DWA，保守策略）
  max_vel_x_backwards: 0.2      # 最大后退速度 0.2 m/s
  max_vel_theta: 1.0            # 最大角速度 1.0 rad/s
  acc_lim_x: 0.5                # 线加速度 0.5 m/s²（显著低于 DWA 的 2.5）
  acc_lim_theta: 0.5            # 角加速度 0.5 rad/s²
  min_turning_radius: 0.0       # 差速驱动可原地转向

  # ---- 障碍约束 ----
  min_obstacle_dist: 0.10       # 障碍物最小距离硬约束 10cm
  inflation_dist: 0.20          # 膨胀影响范围 20cm
  include_costmap_obstacles: true

  # ---- 优化权重 ----
  weight_max_vel_x: 2           # 线速度约束权重
  weight_max_vel_theta: 1       # 角速度约束权重
  weight_obstacle: 50           # 障碍物避让权重
  weight_kinematics_nh: 1000    # 非完整约束权重（差速驱动）
  weight_optimaltime: 1         # 时间最优权重
```

两种局部规划器的系统性对比如表5-2所示：

| 对比维度 | DWA | TEB |
|:---|:---|:---|
| 算法原理 | 速度空间采样 + 多目标评价函数 | 弹性带轨迹优化（非线性最小二乘） |
| 算法复杂度 | $O(n_v \cdot n_\omega)$（采样数之积） | $O(k^3)$（$k$ 为弹性带节点数） |
| 最大线速度 | 0.25 m/s | 0.20 m/s |
| 线加速度限制 | 2.5 m/s² | 0.5 m/s²（更保守） |
| 障碍物最小距离 | 软约束（`occdist_scale` 权重） | 硬约束（`min_obstacle_dist: 0.10m`） |
| 后退能力 | 支持（`min_vel_x: -0.25`） | 支持（`allow_init_with_backwards_motion: true`） |
| 窄通道通行 | 较好（高采样密度覆盖） | 优秀（弹性带可连续变形穿越） |
| 路径平滑度 | 良好（离散速度选择） | 优秀（连续优化含平滑约束） |
| 参数调节难度 | 中等（3个评分权重） | 较高（多个优化权重） |
| 典型 CPU 占用 | 约 8~15%（香橙派） | 约 12~20%（香橙派） |

**表5-2 DWA 与 TEB 局部规划器对比**

本系统默认启用 TEB 局部规划器。TEB 的弹性带优化能够在窄走廊中生成更平滑的轨迹，其 `min_obstacle_dist: 0.10m` 的硬约束提供了确定性的安全保障。而 DWA 作为备选方案保留，其计算量较低、参数调节直观，适合在对 CPU 资源敏感的场景中使用。

## 5.5 恢复行为配置

当 `move_base` 导航失败时（全局路径规划失败、局部规划器无法生成安全速度、控制器超时等），框架按优先级依次触发恢复行为，尝试解除困境后重新规划。本系统的恢复行为配置来自 `move_base_params.yaml`：

```yaml
# move_base_params.yaml
controller_frequency: 2.0        # 控制循环频率 2.0Hz（每 500ms 调用一次局部规划器）
controller_patience: 15.0        # 控制器容忍时间 15s
planner_frequency: 0.2           # 全局重规划频率 0.2Hz（每 5s 重规划一次）
planner_patience: 5.0            # 规划器容忍时间 5s
conservative_reset_dist: 3.0     # 保守重置距离 3.0m
oscillation_timeout: 10.0        # 振荡检测超时 10s
oscillation_distance: 0.2        # 振荡检测位移阈值 0.2m
clearing_rotation_allowed: false  # 禁用原地旋转恢复
```

恢复行为触发流程与本系统配置如表5-3所示：

| 恢复行为 | 触发条件 | 动作描述 | 本系统配置 |
|:---|:---|:---|:---|
| 代价地图清除 | 首次规划失败 | 清除 `conservative_reset_dist`（3.0m）范围外的代价地图障碍标记 | 启用 |
| 原地旋转 | 清除后仍失败 | 原地旋转以获取新的激光观测，清除"幽灵"障碍 | **已禁用** |
| 再次清除+重规划 | 旋转后仍失败 | 使用清除后的代价地图重新全局规划 | 启用 |
| 导航放弃 | 多次恢复后仍失败 | 放弃当前目标，报告 `ABORTED` 状态 | `oscillation_timeout: 10.0s` |

**表5-3 恢复行为配置**

**关键设计决策**：`clearing_rotation_allowed: false` 禁用了原地旋转恢复行为。这一决策的原因是：本系统差速底盘在原地旋转时，编码器积分的航向误差较大，可能导致 AMCL 定位产生偏移，在宽度有限的走廊中存在碰撞墙壁的风险。禁用旋转后，`move_base` 在遇到障碍时主要依赖代价地图清除和全局重规划，辅以 `oscillation_timeout: 10.0s` 的振荡检测机制——当机器人在 10 秒内反复移动不超过 0.2m 时判定为振荡，放弃当前目标并通过 ZMQ 向树莓派5报告导航失败，由 `waypoint_patrol` 决定跳过该巡检点继续下一个。

`controller_frequency: 2.0Hz` 使控制周期为 500ms，这一较低的控制频率适配香橙派3LTS 的算力约束——在同时运行底盘控制、雷达驱动、AMCL、`move_base` 和 ZMQ 桥接的情况下，2Hz 控制频率将 `move_base` 的 CPU 占用控制在合理范围。`planner_frequency: 0.2Hz` 表示全局路径每 5 秒重规划一次，在以静态障碍物为主的室内场景中已足够应对环境变化。

## 5.6 多巡检点任务调度设计

### 5.6.1 任务调度架构

多巡检点任务调度由树莓派5端的 `waypoint_patrol` 节点实现，该节点运行于 ROS2 Jazzy 环境中。与直接使用 ROS2 Nav2 导航不同，本系统通过 ZMQ 消息桥接与香橙派端的 `zmq_nav_bridge` 节点通信，间接驱动 ROS1 `move_base` 执行导航任务。

`waypoint_patrol` 采用 `MultiThreadedExecutor` + 独立巡逻线程的架构设计。ROS2 中若在回调函数内调用 `spin_until_future_complete` 会导致单线程执行器死锁，因此巡逻主循环在独立 `threading.Thread` 中运行，由 `MultiThreadedExecutor`（双线程）在主线程中处理 ROS2 回调和 Action 反馈：

```python
class WaypointPatrol(Node):
    def __init__(self):
        super().__init__('waypoint_patrol')
        # ZMQ 桥接参数
        self.declare_parameter('use_zmq_bridge', False)
        self.declare_parameter('zmq_goal_addr', 'tcp://192.168.50.2:5555')
        self.declare_parameter('zmq_status_addr', 'tcp://192.168.50.2:5556')

        if self.use_zmq_bridge:
            # PUSH 套接字发送导航目标
            self._zmq_goal_socket = self._zmq_context.socket(zmq.PUSH)
            self._zmq_goal_socket.connect(self.zmq_goal_addr)
            # SUB 套接字接收导航结果
            self._zmq_status_socket = self._zmq_context.socket(zmq.SUB)
            self._zmq_status_socket.connect(self.zmq_status_addr)

        # 启动巡逻线程（独立于 executor spin，避免死锁）
        self._patrol_thread = threading.Thread(
            target=self._patrol_worker, daemon=True)
        self._patrol_thread.start()
```

该节点同时保留了通过 DDS 直连 Nav2 `NavigateToPose` Action 的能力（`use_zmq_bridge: false` 时），便于未来向纯 ROS2 架构迁移。

### 5.6.2 ZMQ 跨板导航目标传递

导航目标以 JSON 消息格式通过 ZMQ PUSH/PULL 套接字传递，导航结果通过 PUB/SUB 套接字回传。消息交互流程如下：

**(1) 树莓派5→香橙派：发送导航目标**

```python
def _navigate_to_zmq(self, x, y, yaw):
    payload = {
        'type': 'nav_goal',
        'goal_id': f'wp_{self._zmq_goal_seq}_{int(time.time()*1000)}',
        'x': x, 'y': y, 'yaw': yaw,
        'frame_id': 'map',
        'timestamp': time.time(),
    }
    self._zmq_goal_socket.send_string(json.dumps(payload))
```

**(2) 香橙派端：接收目标并转发至 move_base**

`zmq_nav_bridge` 节点运行于 ROS1 环境，通过 ZMQ PULL 套接字接收 `nav_goal` JSON 消息，将其转换为 ROS1 `PoseStamped` 消息发布至 `/move_base_simple/goal` 话题：

```python
class ZmqNavBridge:
    def _recv_loop(self):
        raw = self._goal_pull.recv_string()
        msg = json.loads(raw)
        pose = PoseStamped()
        pose.header.frame_id = msg.get('frame_id', 'map')
        pose.pose.position.x = float(msg['x'])
        pose.pose.position.y = float(msg['y'])
        qz, qw = _yaw_to_quat(float(msg['yaw']))
        pose.pose.orientation.z = qz
        pose.pose.orientation.w = qw
        self._goal_pub.publish(pose)
```

**(3) 香橙派→树莓派5：回传导航结果**

`zmq_nav_bridge` 订阅 `/move_base/result`（`MoveBaseActionResult` 类型），将导航状态编码为 JSON 并通过 ZMQ PUB 套接字广播：

```python
    def _result_cb(self, msg):
        payload = {
            'type': 'nav_result',
            'goal_id': self._last_goal_id,
            'status': int(msg.status.status),
            'text': msg.status.text,
            'timestamp': time.time(),
        }
        self._status_pub.send_string(json.dumps(payload))
```

树莓派5端 `waypoint_patrol` 通过 ZMQ SUB 套接字接收 `nav_result`，根据 `goal_id` 匹配和 `status` 字段判断导航是否成功（`GoalStatus.STATUS_SUCCEEDED = 3`），带有 `nav_timeout: 120s` 的超时保护。

### 5.6.3 巡检点配置与巡逻策略

巡检点通过 YAML 文件配置（`waypoints.yaml`），每个巡检点指定在 `map` 坐标系下的位置和到达后的朝向：

# waypoints.yaml

### 5.7.3 局部规划器路径跟踪特性分析

通过记录 AMCL 发布的 `/amcl_pose` 话题（100ms 采样一次）与 `move_base` 的全局规划路径话题，计算每时刻机器人位姿到最近规划路径点的垂直距离（横向偏差）并进行分析：

**直线段跟踪**：在走廊直线段中，DWA 的高路径跟踪权重（`path_distance_bias: 32.0` 远大于 `occdist_scale: 0.02`）使机器人紧密跟踪全局路径中心线；TEB 的弹性带优化同样将路径偏差作为代价项最小化。两者在直线段的横向偏差主要由 AMCL 定位噪声引起（约 2~4cm），局部规划器自身的控制偏差可忽略。

**弯道段特性**：在 T 形路口转弯时，DWA 通过在速度空间中搜索大角速度-小线速度的组合实现转弯，`min_vel_trans: 0.13 m/s` 保证弯道中机器人不会完全停止。TEB 在弯道处表现更平滑——弹性带优化自然地将转弯轨迹调整为圆弧形，避免了 DWA 可能出现的离散速度切换抖动。`sim_time: 2.0s`（DWA）和 `max_global_plan_lookahead_dist: 3.0m`（TEB）的前瞻设置使两种规划器都能提前感知弯道走向，避免过度转向。

**障碍物绕行**：场景B中机器人接近静态纸箱障碍物时，`obstacle_range: 2.5m` 内的激光观测将障碍物标记在局部代价地图中。DWA 在轨迹评分时自动淘汰碰撞轨迹并选择绕行路径；TEB 的 `min_obstacle_dist: 0.10m` 硬约束确保弹性带始终与障碍物保持至少 10cm 间距。由于公共参数中 `inflation_radius: 0.1m` 较小，机器人可以较近距离通过障碍物旁侧，适合宽度有限的走廊场景。

## 5.8 本章小结

本章详细阐述了基于 ROS1 `move_base` 框架的自主导航系统设计与实现。从框架架构出发，介绍了 `move_base` 的双层规划与恢复行为调度机制，以及导航闭环在香橙派本地完成的设计优势；深入分析了 Costmap2D 的分层代价模型与膨胀代价公式，结合 `costmap_common_params.yaml` 给出公共参数配置（`inflation_radius: 0.1m`、`cost_scaling_factor: 10.0` 的紧凑膨胀策略）及全局（1Hz、VoxelLayer）与局部（5Hz、3m×3m 滚动窗口）代价地图的完整参数；推导了 GlobalPlanner 的 Dijkstra/A* 全局规划算法与代价缩放机制；重点分析了 DWA 局部规划器的动态窗口速度采样、前向仿真和多目标评价函数（`path_distance_bias: 32.0`、`goal_distance_bias: 20.0`、`occdist_scale: 0.02`），并与 TEB 弹性带优化规划器进行了算法原理、参数配置和性能指标的系统对比；设计了基于 ZMQ 跨板桥接的 `waypoint_patrol` 巡检点巡航节点，采用 MultiThreadedExecutor + 独立巡逻线程避免死锁，通过 JSON 消息交互驱动 ROS1 `move_base` 导航并实现导航失败的自动跳过机制；三场景导航实验从导航成功率、到达时间、路径跟踪偏差和恢复行为触发等维度验证了系统的导航性能。
