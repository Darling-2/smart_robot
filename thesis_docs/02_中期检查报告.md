## 一、课题研究进展总结

### 1.1 总体进展概述

经过前半学期的研究与开发，本课题已按计划完成了异构硬件平台搭建、ROS1/ROS2 环境部署与**以太网直连跨板通信（ZMQ 业务消息）**验证、ROS1 侧激光 SLAM 建图与 AMCL 定位调试、ROS1 侧 move_base 自主导航配置等核心工作；ROS2 侧视觉检测模块正在推进数据集制作与模型训练。具体进展如下。

---

### 1.2 阶段一：硬件集成与通信架构验证（第1–4周，已完成）

#### 1.2.1 双板硬件平台集成

以 **NanoRobot 差速驱动底盘** 为载体，完成了以下硬件集成：

| 硬件组件 | 配置情况 |
|:---|:---|
| 香橙派3LTS | Ubuntu 20.04 + ROS1 Noetic |
| 树莓派5（4GB） | Ubuntu 24.04 + ROS2 Jazzy |
| STM32底盘控制板 | USB串口连接香橙派，运行底盘控制脚本完成串口协议收发 |
| EAI激光雷达 | 连接香橙派，通过 ROS1 雷达驱动节点发布激光扫描数据 |
| 720p USB摄像头 | 连接树莓派5，通过 ROS2 相机驱动节点发布图像数据 |
| 百兆以太网 | 两板直连，静态IP配置（香橙派：192.168.50.2，树莓派5：192.168.50.3） |

#### 1.2.2 ROS1 底盘控制调试

香橙派3LTS 上的底盘控制节点通过 USB 串口与 STM32 进行通信，订阅速度指令话题并转换为差速轮速指令，同时读取编码器数据发布里程计话题。使用键盘遥控工具测试验证底盘运动正常，2米直线里程计累积误差约 2.5 cm。

ROS1 端同时发布激光雷达安装偏置的静态 TF 变换，供后续 TF 链使用。

#### 1.2.3 跨板通信桥接部署与验证

在香橙派3LTS（ROS1端）与树莓派5（ROS2端）之间通过百兆以太网直连（192.168.50.2↔192.168.50.3）部署跨板通信模块。本课题最终采用“**业务消息跨板**”的思路：

- **ROS1 侧**保持导航闭环（底盘控制、里程计、激光、SLAM/定位、move_base）本地运行，不对激光扫描、里程计、速度指令等高频话题进行跨板透明转发；
- **ROS2 侧**仅负责巡检任务调度与视觉检测，通过 ZMQ 在以太网上与 ROS1 侧交换少量关键业务消息。

经验证，跨板通信实现以下两类业务消息的可靠收发（JSON 编码）：

| 业务消息 | 方向 | 主要字段（示例） | 指标 |
|:---|:---:|:---|:---|
| 导航目标 | ROS2→ROS1 | 目标编号、坐标、航向角、坐标系、时间戳 | 毫秒级往返时延 |
| 导航结果 | ROS1→ROS2 | 目标编号、状态码、说明文本、时间戳 | 结果回传稳定，多次测试无丢包 |

### 1.3 阶段二：SLAM建图与导航调试（第5–9周，已完成）

#### 1.3.1 GMapping 2D SLAM 建图（ROS1 侧主线）

在香橙派3LTS（ROS1 Noetic）上启动 GMapping 建图节点，订阅本地激光扫描与 TF 链（含里程计变换）进行室内建图。建图完成后使用地图保存工具导出栅格地图文件，用于后续 AMCL 定位与 move_base 导航。

在实验场地（公寓楼道，约 20m×5m 区域）完成建图：
- 手动遥控机器人沿走廊巡航一圈，保持速度平稳、尽量覆盖关键拐角与门洞区域；
- 保存地图文件并在 RViz 中叠加激光点云进行一致性检查；
- 地图分辨率等参数按实际雷达与场地需求配置（填写实测配置）。

#### 1.3.2 AMCL 自主定位配置

在香橙派3LTS（ROS1 Noetic）上依据导航参数文件启动 AMCL 定位：
- 初始位姿通过 RViz 可视化工具设置；
- 通过调整粒子数上下限、激光模型与重采样参数，兼顾定位稳定性与计算开销；
- 在走廊环境下可实现稳定跟随与重新收敛（填写实测指标）。

#### 1.3.3 move_base 自主导航配置（ROS1 侧主线）

在香橙派3LTS（ROS1 Noetic）上配置并启动 move_base 导航栈，形成“定位—规划—控制”的本地闭环：

- **全局规划器**：采用 NavFn 或 GlobalPlanner，生成可行的全局路径；
- **局部规划器**：采用 DWA 局部规划器，根据局部代价地图进行避障与速度控制；
- **代价地图**（局部/全局）：障碍层订阅本地激光扫描数据；机器人外形与膨胀半径依据实测尺寸与走廊宽度配置；
- **恢复行为**：结合旋转、后退与等待等策略，提高在狭窄走廊与动态行人场景下的鲁棒性。

#### 1.3.4 导航功能测试结果

在走廊中设置4个巡检目标点进行导航测试（10次重复）：

| 测试场景 | 成功次数/总次数 | 平均到点时间（5m行程） | 备注 |
|:---:|:---:|:---:|:---|
| 无障碍空旷走廊 | 10/10 | ~52s | 路径平滑，无抖动 |
| 走廊放置静态纸箱 | 10/10 | ~68s | 绕行成功，路径重规划正常 |
| 行人正面相向行走 | 8/10 | ~75s | 2次因行人持续占道触发等待超时，恢复行为后成功 |

局部规划器在走廊直线行驶中路径跟踪平滑，无明显偏摆；接近目标点时自动减速并停止，到点精度满足巡检需求。

说明：上述描述中“控制器/规划器”等名词以本系统主线的 move_base + DWA 为准；若在对比实验中启用 ROS2 Nav2/RPP，将在论文“扩展对比”部分单独呈现参数与结果。

**遇到的问题**：
- 激光扫描数据经跨板传输后，时间戳与系统时间存在微小偏差（约0.05s），导致代价地图更新时 TF 查询偶发超时警告。**解决方案**：在导航参数文件中设置 TF 变换容忍度为 1.0s，消除超时警告。

## 三、遇到的主要问题及解决情况

### 问题1：跨板消息的时间戳对齐与日志关联

**现象**：跨板通信采用 ZMQ 业务消息后，两板各自记录时间戳；若系统时钟存在偏差，会影响巡检流程中“派发目标—到点—触发检测—回传结果”的时序统计与问题定位。

**解决方案**：通过 chrony 对两板进行 NTP 对时，将时钟偏差控制在可接受范围内；同时在导航目标与导航结果消息中携带发送端时间戳，并在接收端记录本地接收时间，以便离线计算往返时延与端到端耗时。

### 问题2：动态行人场景下的局部避障与恢复策略

**现象**：走廊行人正面相向或长时间占道时，局部代价地图会持续检测到近距离障碍，机器人可能出现“停—等—再尝试”的循环，导致巡检效率下降。

**解决方案**：在 move_base 的局部规划器与恢复行为中，综合调整停止阈值、采样空间、加速度限制等参数，并配置旋转、后退、等待等恢复策略，使机器人在安全前提下尽快恢复导航或触发重规划。

### 问题3：走廊弱特征环境下的建图/定位漂移

**现象**：公寓楼道环境重复性较高（墙壁特征单调），在转角、长直道等区域可能出现地图局部轻微拉伸或定位短时漂移。

**解决方案**：在 ROS1 侧对 GMapping/AMCL 的关键参数（粒子数、更新阈值、激光模型等）进行调参；同时规范建图轨迹与速度，减少急停急转带来的里程计误差累积。

### 问题4：视觉推理资源占用与导航/检测协同

**现象**：树莓派5在运行相机驱动与 YOLOv8n 推理时会产生一定 CPU/内存占用，若持续推理可能影响任务调度节点的实时性。

**解决方案**：视觉检测采用帧率限制模式（系统限频 5 FPS），通过独立的检测节点持续推理，与巡检调度节点和记录管理节点相互独立；同时对推理模型与输入分辨率进行权衡，确保资源占用不影响整机稳定运行。
