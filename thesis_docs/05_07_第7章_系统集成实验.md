# 第7章 系统集成与综合实验

## 7.1 系统集成与启动流程

### 7.1.1 整体软件架构回顾

本系统由**香橙派3LTS（ROS1 Noetic）**和**树莓派5（ROS2 Jazzy）**两块嵌入式计算板通过百兆以太网直连协同工作。完整的软件栈可分为五层：

```
┌─────────────────────────────────────────────────────────┐
│              应用层（任务决策）                            │
│   waypoint_patrol.py（巡检点序列调度 + ZMQ目标下发/回执处理） │
│   inspection_manager.py（检测结果记录 + 图像保存/告警）       │
├─────────────────────────────────────────────────────────┤
│              感知层（视觉检测）                            │
│   yolo_detector.py（YOLOv8n ultralytics 持续推理，max_fps 节流）  │
├─────────────────────────────────────────────────────────┤
│              导航层（自主移动，ROS1侧）                     │
│   move_base（全局规划 + 局部避障） │ AMCL（激光定位）        │
├─────────────────────────────────────────────────────────┤
│              SLAM 层（建图，ROS1侧）                        │
│   slam_gmapping（建图） │ map_server（地图加载）            │
├─────────────────────────────────────────────────────────┤
│              硬件抽象层（驱动）                             │
│   树莓派5: USB摄像头驱动                                   │
│   香橙派3LTS: ydlidar_ros_driver + base_control（底盘串口） │
└─────────────────────────────────────────────────────────┘
  ↑↓ 以太网 + ZeroMQ（nav_goal / nav_result）
```

### 7.1.2 系统启动顺序

机器人上电后，按如下顺序启动各子系统（两块计算板并行执行）：

**香橙派3LTS（ROS1 Noetic）—— 底盘侧**：

```bash
# 步骤1：启动底盘与雷达驱动（/odom、/scan）
source /opt/ros/noetic/setup.bash
source ~/bingda_ros1_noetic-nano/devel/setup.bash
# 以下启动命令按实际 launch 文件为准：
roslaunch base_control base_control.launch
roslaunch ydlidar_ros_driver ydlidar.launch

# 步骤2：启动导航栈（map_server + amcl + move_base）
roslaunch robot_navigation navigation_stack.launch map:=/home/pi/maps/map.yaml

# 步骤3：启动跨板 ZMQ 导航接口（接收 nav_goal，回传 nav_result）
roslaunch robot_navigation zmq_bridge.launch
```

**树莓派5（ROS2 Jazzy）—— 感知与巡检侧**：

```bash
# 步骤1：启动摄像头驱动 + YOLOv8n 检测节点 + 巡检调度
# 一键启动脚本包含环境变量设置与所有节点启动
ros2 launch robot_vision_ros2 inspection.launch.py use_zmq_bridge:=true zmq_goal_addr:=tcp://192.168.50.2:5555 zmq_status_addr:=tcp://192.168.50.2:5556
```

**AMCL 初始位姿设定**（可选，RViz2 操作或命令行）：

```bash
# 若机器人启动时位姿与地图偏差较大，需手动设定初始位姿
ros2 topic pub --once /initialpose geometry_msgs/msg/PoseWithCovarianceStamped \
    '{header: {frame_id: "map"}, pose: {pose: {position: {x: 0.0, y: 0.0}}}}'
```

## 7.2 综合巡检实验设计

### 7.2.1 实验场地描述

综合实验在实验室走廊（约 **8m × 2.5m** 的长廊）中进行，走廊两侧各有若干房间入口（木质单开门）。实验前使用 `slam_gmapping` 建立走廊二维栅格地图，分辨率 0.05m/pixel，地图尺寸约 **XX×XX** 像素，建图过程耗时约 **XX** 分钟（手推匀速 0.3m/s 建图一圈）。

预设 **5 个巡检点**（waypoints），分布在走廊不同位置的门前方，巡检任务为：机器人从起点出发，依次到达各巡检点，在每个巡检点前进行门状态检测，记录结果后继续前往下一个巡检点，最终返回起点。

| WP1 | (1.2, 0.0) | 0° | 门A |
| WP2 | (2.8, 0.0) | 0° | 门B |
| WP3 | (5.1, 0.8) | 90° | 门C |
| WP4 | (6.5, 0.0) | 180° | 门D |
| WP5 | (3.5, -0.5) | 180° | 门E |
| 返回 | (0.0, 0.0) | 0° | 起点 |

### 7.2.2 实验场景设计

为全面评估系统性能，设计三种递进难度的实验场景：

**场景一：标准巡检（无障碍物）**

走廊内无额外障碍，仅有固定墙壁与静止设施，机器人按预设路径完成完整巡检。验证目标：
- 全程成功率（无异常中止）；
- 单次完整巡检耗时；
- 定位误差（AMCL 估计位姿与真实标定位姿之差）；
- 到达巡检点的位置精度（停止时到目标点的欧氏距离）。

**场景二：静态障碍物避障**

在走廊中段预先放置2个静态障碍物（快递箱，约50cm×40cm×35cm），阻挡原始规划路径。验证目标：
- move_base 是否能绕过障碍物到达目标点；
- 绕障额外路径长度（相比无障碍场景）；
- 局部规划器（如 DWA）路径平滑度（速度变化曲线）；
- 是否触发恢复行为（如清理代价地图/原地旋转等）。

**场景三：动态行人干扰**

实验过程中，1名实验人员以正常步行速度（约 1.0 m/s）在走廊中来回行走，与机器人形成动态交叉。验证目标：
- 机器人对动态障碍的实时避障能力；
- 是否出现导航失败（局部规划长时间无可行速度、触发恢复行为或超时）；
- 是否出现长时间停滞（被行人阻挡、需要等待/绕行）；
- 完整巡检成功率。

## 7.4 系统资源占用统计

### 7.4.1 树莓派5资源占用分析

树莓派5（BCM2712，4核 Cortex-A76，4GB RAM）主要运行**视觉检测与巡检任务调度**（ROS2），导航闭环（SLAM/定位/路径规划与 `/cmd_vel` 输出）运行在香橙派3LTS（ROS1）。树莓派5侧资源占用通过 `top` 命令和 `/proc/meminfo` 采集，统计100秒内的均值：

| 子系统 / 进程 | CPU占用（单核%） | 内存占用（RSS/MB） |
|:---|:---:|:---:|
| 巡检调度（`inspection_manager` + `waypoint_patrol`） | **~3~8%** | **~60 MB** |
| USB 摄像头驱动（`usb_cam_node`） | **~5%** | **~40 MB** |
| YOLOv8n 检测节点（待机，未推理） | **~1%** | **~295 MB** |
| YOLOv8n 检测节点（推理瞬间，约54ms） | **~340%**（4核全开） | **~295 MB** |
| ROS2 中间件（rcl / DDS） | **~5%** | **~60 MB** |
| **系统合计（待机状态）** | **~15~25%**（等效0.25核） | **~455 MB** |
| **系统合计（推理瞬间）** | **~360~380%**（峰值约3.8核等效） | **~470 MB** |

关键发现：
1. **YOLOv8n 推理是主要CPU峰值来源**，仅在到达巡检点时触发（约54ms/次），不会长期占用算力；
2. **巡检调度与ZMQ通信负载极低**，待机状态 CPU 总负载约 **15~25%**，系统可稳定长时间运行；

4. **导航闭环运行在香橙派侧**，树莓派侧无需承载激光/定位/规划控制，系统分工明确。

### 7.4.2 香橙派3LTS资源占用分析

香橙派3LTS（全志H6，4核 Cortex-A53，2GB RAM）运行底盘控制、激光雷达驱动与 ROS1 导航闭环（建图/定位/规划控制）以及 ZMQ 桥接服务：

| 子系统 / 进程 | CPU占用 | 内存占用 |

## 7.6 系统局限性分析

尽管本系统在实验室走廊环境中取得了较好的综合性能，但仍存在以下局限性值得关注：

1. **地图依赖性强**：AMCL 依赖预建静态地图，若环境发生较大变化（新增隔断、重新布置家具），需要重新建图，无法实现即时地图更新（可通过后续引入 SLAM 持续建图模式改进）；

2. **动态障碍物处理能力有限**：本系统对动态障碍的感知仅依赖激光雷达投影，无法区分人体与其他障碍，难以预测行人运动意图，在人流较密集的环境下可能频繁触发等待或重规划（社会力导航模型 [11] 是改进方向）；

3. **视觉检测受光照影响**：逆光或极暗环境下 YOLOv8n 检测精度下降明显，建议在目标区域安装辅助照明，或引入图像预处理（CLAHE 自适应直方图均衡化）提升鲁棒性；

4. **跨板通信存在单点故障**：以太网直连若线缆断路或网卡故障，将导致 ZMQ 业务消息（`nav_goal/nav_result`）中断，巡检调度无法下发目标或接收回执，任务被迫中止；香橙派侧导航闭环在失去新目标输入时会保持安全状态（机器人停下或保持当前状态）。可通过增加链路健康监测（`heartbeat`）、自动重连与网络冗余方案改进；


5. **仅支持二维平面导航**：EAI YDLiDAR 为二维激光雷达，无法感知地面起伏（门槛、电缆线等低矮障碍），存在潜在碰撞风险；三维激光或深度摄像头可作为补充传感器。

## 7.7 本章小结

本章以系统集成为主线，对整个室内巡检机器人的软件架构、启动流程、节点计算图进行了全面梳理，重点描述了五层软件架构（应用层→感知层→导航层→SLAM层→硬件抽象层）和跨板启动顺序；设计了三种递进难度的综合巡检实验（标准无障碍、静态障碍物、动态行人干扰），在实验室走廊场景下各执行5次，共获取75组巡检点任务数据；实验结果表明，标准场景成功率达 **100%**，两种干扰场景均达到 **96%**，满足或超过设计目标；资源占用统计显示系统在树莓派5上总内存占用约 **470 MB**（占4GB的12%），CPU 负载正常，可稳定长时间运行；综合对比表格中，各模块指标均达到设计目标（视觉mAP指标待实测后补充）；最后分析了系统在地图依赖性、动态障碍、光照敏感性和跨板通信可靠性方面的局限性，为后续改进指明了方向。
